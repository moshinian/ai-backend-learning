# 结算单电子盖章

## 1. 项目背景

云服务结算系统需要把结算单 PDF 提交至外部盖章中心，并在盖章完成后将文件归档到 EDM 文档平台，供业务人员下载和后续管理。

该模块由本人独立负责方案设计与开发，涉及本系统、盖章中心和 EDM 三方系统之间的文件传输、异步回调、状态流转和失败恢复。

## 2. 解决的问题

1. 将原本需要人工处理的结算单盖章动作接入线上流程。
2. 支持业务人员批量申请盖章，单个后端请求最多处理 20 份结算单；实际操作中可能连续处理约 10 页，累计约 200 份。
3. 解决跨系统异步处理中的请求关联、重复回调、超时未回调和单笔失败隔离问题。
4. 同时保留原始 PDF 与盖章后 PDF，使业务可以下载盖章版本，也可以将结算单恢复为未盖章版本。
5. 功能上线运行 3 个月，期间未发生生产故障或数据错误。

## 3. 核心链路

### 3.1 申请盖章

1. 用户在页面勾选最多 20 份结算单并提交申请。
2. 系统按结算单号逐笔获取 Redis 锁，并通过带原状态条件的数据库 `UPDATE` 将结算单从未盖章或盖章失败更新为盖章中。
3. 系统为本次申请生成并持久化唯一请求标识，将单笔任务提交到自定义线程池；调度代码提交完成后即释放 Redis 锁。
4. 后台任务先调用盖章中心的 PDF 上传接口，成功后取得 `docId`。
5. 后台任务再携带 `docId`、盖章类型、盖章模式和盖章位置等参数调用盖章申请接口。
6. 单笔“上传 PDF + 发起盖章”通常约耗时 2 秒；任一步骤异常时，将该结算单更新为盖章失败并记录失败原因，不影响同批其他结算单。

### 3.2 盖章结果回写

1. 盖章中心内部异步完成盖章后，调用云服务结算系统提供的 Spring Web 回调接口，并传入 `fileUrl`、`docId` 和可定位原申请的唯一标识。
2. 回调接口按唯一请求标识或 `docId` 找到对应结算单和本次盖章申请。
3. 系统从 `fileUrl` 下载盖章后的 PDF，再上传至 EDM。
4. EDM 上传完成后，通过带当前状态和本次申请标识条件的原子 `UPDATE` 将结算单更新为盖章成功，并保存盖章文件的 EDM 地址。
5. 盖章结果回写由 Spring 请求线程处理，不经过申请阶段使用的自定义线程池。

### 3.3 下载、恢复与超时处理

1. 结算单表分别保存原始 PDF 和盖章后 PDF 的 EDM 地址；盖章成功后，用户默认下载盖章版本。
2. 业务可以把当前文件切回原始 PDF，并将状态恢复为未盖章。
3. 对持续处于盖章中超过 1 小时的数据，补偿任务会将其重置为未盖章，并清理当前申请的 `docId`、`fileUrl` 等回写信息，允许用户从头重新发起盖章。
4. 系统与外部服务的请求和响应保留在全局 API 日志表中；结算单主表只保留最近一次操作人的审计字段。

## 4. 关键设计

### 4.1 短锁抢占与持久化状态

Redis 锁只保护单个结算单的申请抢占和异步任务投递，不覆盖整个外部盖章周期。锁释放后，数据库中的盖章中状态和本次申请唯一标识继续承担持久化并发控制。

状态更新采用带原状态条件的原子 `UPDATE`，避免同一结算单被并发重复申请。Redis 锁负责降低并发碰撞，数据库状态机负责最终处理权，两者职责不同。

### 4.2 有界线程池与单笔失败隔离

申请阶段使用核心线程数 8、最大线程数 16、有界队列 100 的线程池。单个请求最多提交 20 笔任务，业务连续翻页操作时可累计约 200 笔，任务会随处理过程逐步出队。

`ThreadPoolExecutor` 会先使用 8 个核心线程，再把任务放入长度为 100 的队列，队列满后才继续扩容到 16 个线程，因此同一时刻大约可容纳 116 笔运行中或待执行任务。线程池沿用默认拒绝策略 `AbortPolicy`；极端快速连续提交导致容量耗尽时，拒绝异常会进入业务异常处理，将对应结算单标记为盖章失败，避免静默丢失，后续可由业务重新发起。

### 4.3 回调关联与并发幂等

每次盖章申请都有持久化的唯一请求标识，并与结算单、`docId` 建立关联。回调更新同时校验当前状态和本次申请标识：

1. 已经盖章成功的记录不会被重复更新。
2. 超时重置后重新发起的申请拥有新的请求标识，旧请求的迟到回调不能覆盖新申请。
3. 数据库条件更新决定最终只有当前有效申请可以将结算单更新为盖章成功。

### 4.4 可恢复状态机

主流程可以概括为：

`未盖章 / 盖章失败 -> 盖章中 -> 盖章成功 / 盖章失败`

盖章失败和超过 1 小时未完成的任务都可以回到可重新申请的状态。该设计没有要求在原中断点继续执行，而是通过“清理当前申请上下文 + 从头重试”恢复，符合该业务低频、单笔处理成本较低的特点。

### 4.5 文件版本分离

原始 PDF 与盖章 PDF 使用独立字段保存，不通过覆盖同一地址实现版本切换。这既保留了原件，也让“恢复未盖章”成为受控的业务状态变更，而不是重新生成或找回文件。

## 5. 边界和风险

以下是现有方案需要如实保留的边界，不作为已经落地的能力写入简历：

1. 回调当前先下载文件并上传 EDM，再执行数据库条件更新。两个重复回调并发到达时，虽然只有一次状态更新能够成功，但仍可能发生重复下载和重复上传。可在文件处理前增加“回调处理中”状态抢占、请求级锁，或使用确定性的 EDM 存储键。
2. EDM 上传成功而数据库回写失败时，重试可能产生孤立文件或重复文件。可补充 EDM 文件幂等键、上传记录和定期清理机制。
3. 回调中的文件下载和 EDM 上传运行在 Spring 请求线程上。当前业务量较低时可以接受，但仍需配置连接、读取和接口超时；若回调量或文件体积上升，应改为快速验签入库后异步处理，并增加队列背压。
4. 超时任务直接重置并重新申请，没有先查询盖章中心的远端处理状态，可能在外部保留一个已经完成或仍在处理的旧任务。当前依靠请求版本校验拒绝迟到回调，但远端任务和文件仍需通过对账或人工协查处理。
5. 全局 API 日志保存外部请求和响应时，需要控制敏感字段脱敏、响应体大小和保留周期。
6. 如果数据库本身不可用，业务失败原因也可能无法写回结算单表，因此仍需依赖应用日志和监控告警定位，不能把“异常后回写失败原因”描述为绝对保证。

## 6. 面试表达入口

### 6.1 一分钟版本

我独立设计并开发过结算单电子盖章模块，对接外部盖章中心和 EDM 文档平台。申请阶段通过两个同步接口完成 PDF 上传和盖章任务提交，但在本系统内用有界线程池逐笔异步执行，单个后端请求最多 20 笔。每次申请都会持久化唯一请求标识和 `docId`，盖章中心完成后通过 Spring 接口异步回调，本系统下载文件、上传 EDM，再通过带状态和申请标识条件的原子更新完成回写。系统还支持失败重试、超过 1 小时任务重置、条件回写防止重复或迟到回调覆盖，以及原始和盖章文件版本切换；上线运行 3 个月未发生生产故障或数据错误。

### 6.2 重点追问入口

1. 为什么 Redis 锁只覆盖任务抢占，不覆盖整个盖章周期？
2. 数据库条件更新如何防止重复申请和旧回调覆盖新申请？
3. 为什么申请阶段用线程池，而回调阶段由 Spring Web 请求处理？
4. 线程池队列满后如何避免任务静默丢失？
5. 超时重置为什么采用从头重试，而不是断点续跑？
6. 重复回调为什么仍可能造成 EDM 重复文件，如何进一步优化？
