# 云服务结算系统：交易流水处理与核销业务域

## 1. 项目背景

云服务结算系统承接上游 FFP 提供的账单文件，将大文件解析为交易流水，完成流水集成、场景处理、订单匹配、辅账生成、ERP 入账、核销对账和异常补偿。

本人从早期接口和场景分支开发逐步成为交易流水场景的主要负责人，后期长期负责交易流水处理与核销业务域，覆盖流水集成、批次处理、分支规则、辅账收敛、ERP 推送与回写、待对账池和核销策略等代码。

上游和下游：

1. 上游是 FFP，提供账单大文件；业务量增长后，文件形态从单个 CSV 演进为多个 CSV 依次解析入库。
2. 下游是 ERP，系统生成标准收据类辅账并推送入账，再接收结果回写。
3. 2C 订单按订单维度核销，2B 订单按结算单维度核销。

## 2. 解决的问题

1. 保证流水从文件集成、批次处理到 ERP 推送过程不重不漏。
2. 根据业务场景正确匹配支付订单、合同、客户和会计科目。
3. 让流水和订单在待对账池中高效匹配，并正确生成已对账数据及对账辅账。
4. 支持上游历史流水纠错，同时保持 ERP 入账和核销链路最终一致。
5. 在数据量增长后控制数据库扫描、网络往返和定时任务执行时长。

## 3. 核心链路

```text
FFP 文件
  -> CSV 解析与接口表入库
  -> 流水批次处理
  -> 业务场景与规则匹配
  -> 按支付流水号匹配 2C / 2B 订单
  -> 获取合同、客户和科目信息
  -> 流水辅账生成
  -> ERP 推送、入账和结果回写
  -> 流水进入收据未对账池

订单 / 发票
  -> 发票未对账池

收据未对账池 + 发票未对账池
  -> 匹配核销
  -> 已对账池
  -> 按发票、收据关系收敛对账辅账
  -> ERP 推送、入账和结果回写
```

## 4. 关键设计

### 4.1 上游历史流水纠错与冲销管道

系统早期已有部分冲销能力。本人承接后完成不同业务场景的适配和缺陷修复，后期相关代码基本由本人维护。

核心机制：

1. 数据批次使用 `C` 表示创建、`U` 表示更新；更新批次携带原批次号，用于持续关联最初批次。
2. 根据批次状态判断原流水已经推进到哪个阶段，并决定补偿动作。
3. 原流水如果仍停留在接口表，则失效原批次流水，新批次正常入库处理。
4. 原流水辅账如果已经在 ERP 入账，则生成科目方向相反、金额为负的流水冲销辅账。
5. 原流水如果已经参与核销，且对账辅账已经在 ERP 入账，则生成撤核销辅账。
6. 撤核销辅账在 ERP 入账成功后，原订单或发票才重新回到发票未对账池，等待新流水重新核销。
7. 原批次冲销辅账与新流水辅账使用不同收据号，作为两笔独立标准收据推送 ERP；两边最终都入账成功后，纠错闭环成立。
8. 任务失败通常通过重试恢复，下游具备幂等能力；持续失败时与 ERP 运维共同定位。

该能力在约三年内支撑过不少于 50 次上游历史流水纠错，能够覆盖数据缺失、内容修正及相同批次重推等情况。

事实边界：

1. 冲销管道不是本人从零设计，早期已有部分实现。
2. 本人的主要贡献是后续场景适配、缺陷修复和长期维护。
3. ERP 幂等能力属于下游能力，不表述为本人建设。

### 4.2 流水匹配支付订单的批处理优化

问题现象：

1. 新业务场景和数据量增长后，待处理流水达到几十万笔。
2. 原实现每分钟处理不到 1000 笔，定时任务严重超时。
3. MySQL 慢 SQL 记录阈值为 1 秒，但慢 SQL 历史中没有对应记录。

定位过程：

1. 最初排查方向是单条慢 SQL 和索引。
2. 检查代码后发现，程序在循环中为每笔流水单独查询一次支付订单。
3. 单条 SQL 都没有超过慢日志阈值，但大量数据库往返形成了典型的 N+1 查询问题。
4. 支付流水号索引原本已经存在；问题不在缺少索引，而在访问方式没有批量化。

优化方案：

1. 先按批次归并流水，每 1000 笔构造一次支付流水号条件。
2. 批量查询对应支付订单，单批查询耗时控制在 1 秒以内。
3. 将订单结果按支付流水号转换为 `Map`，随后在内存中完成流水与订单匹配。
4. 批次大小 1000 来自既有批处理经验和测试结果，满足业务基线后上线，并非理论上的固定最优值。

结果：

1. 整体处理能力达到 100 万笔 8 分钟以内的项目基线。
2. 该案例的核心收益来自减少数据库往返，而不是单纯新增索引。

另一个增长问题是部分订单需要半年甚至一年后才能核销，导致全局核销扫描范围逐月扩大。处理方式是将全局核销任务按业务月份拆成小批次执行，减少单次查询需要扫描的数据范围。由于当前未确认物理分区表细节，对外只表述为“按月份拆分任务和查询范围”。

### 4.3 对账辅账重复与漏生成事故

问题现象：

1. 用户在推送对账辅账时收到 ERP 原收据余额不足的反馈。
2. 数据排查发现，部分对账辅账被重复生成；同时存在上层状态显示已生成、实际辅账没有生成的情况。
3. 多个数据批次受到影响，涉及数千笔对账辅账及不下百万条关联明细凭证。

排查过程：

1. 先根据批次、收据和辅账数据确认影响范围。
2. 翻查任务日志，发现大量 SQL 锁等待超时和 JDBC 回滚异常。
3. 与同事共同检查代码，发现预期负责状态更新和辅账生成的事务方法存在同类内部自调用。
4. 系统此前从女娲框架迁移到自研 Truss 框架，迁移可能暴露了历史上不规范的事务注解写法，但目前没有证据将责任归因于某位同事或单一迁移动作。

技术判断：

1. 数据库锁等待超时是故障触发条件之一。
2. Service 内部自调用不会经过默认的 Spring 事务代理，因此状态更新和辅账生成没有被可靠纳入预期的统一事务边界。
3. `Could not roll back JDBC transaction` 表示 Spring 尝试调用 JDBC `Connection.rollback()` 时又发生异常，只能证明应用当时无法确认回滚结果，不能单独证明数据库一定部分提交。
4. 状态和结果不一致的关键仍是业务步骤没有被可靠地统一提交或回滚；回滚异常属于连接层或数据库异常状态的补充证据。

修复方案：

1. 新建独立的 Spring Service。
2. 将需要保持一致性的状态更新和辅账生成方法迁移到该 Service。
3. 在对外调用的方法上增加事务注解，使调用真正经过 Spring 事务代理。
4. 在测试环节验证异常情况下的事务一致性。
5. 修复后没有再发生相同问题。

事实边界：

1. 事故定位和代码检查由本人和同事协作完成，不表述为单人完成全部处置。
2. 当前没有完整历史异常栈，不能断言 JDBC 回滚失败直接造成了哪一笔脏数据。
3. 当前没有确认历史数据修复的个人职责，不在简历中声称本人完成了全部存量数据修复。

### 4.4 配置驱动的策略模式规则系统

不同业务场景的规则实现放在不同 Java 类中，所有实现类继承同一个规则接口，并由 Spring 容器管理。

规则配置表包含：

1. 规则 ID。
2. 场景 ID。
3. 字段名 1、字段值 1。
4. 字段名 2、字段值 2。
5. 其他用于判断业务场景的配置条件。

运行过程：

1. 加载规则配置数据。
2. 根据当前流水属性执行 `match`，找到符合条件的候选规则。
3. 在候选结果中按照既定顺序选择规则；当前记忆是选择场景 ID 较小的规则，最终口述前仍需确认它是否等同于显式优先级。
4. 根据规则 ID 从规则实现 `Map` 中取得对应的 Spring Bean。
5. 通过统一规则接口调用具体实现，完成场景处理。
6. 规则处理完成后回到公共流水处理和辅账链路。

该结构中：

1. 统一规则接口和不同实现类构成策略模式。
2. 规则 ID 到 Spring Bean 的 `Map` 构成策略注册表，承担按 ID 查找实现的职责。
3. 配置表负责数据型规则匹配，策略类负责需要代码表达的业务行为。
4. 新增规则时可以增加实现类并注册映射；只调整匹配字段时可以修改配置，减少公共主流程中的分支修改。

事实边界：

1. 当前可以确认规则实现类由 Spring 管理并进入 `Map`，但尚未确认 `Map` 是由 Spring 直接注入，还是由启动逻辑遍历 Bean 后构建。
2. 当前尚未确认规则配置是每次从数据库全量读取，还是加载后缓存；面试中不主动声称缓存机制。
3. 如果确实使用场景 ID 大小承担优先级，复盘时应考虑增加显式 `priority` 字段，避免业务优先级与数据库 ID 耦合。

### 4.5 ERP 凭证 ID 命名空间碰撞与撤核销隔离

业务模型：

1. 一笔对账辅账由一张收据和一张发票构成。
2. 收据侧收敛相同收据号下的流水，发票侧收敛相同发票号下的订单。
3. ERP 为收据和发票使用独立的凭证 ID 序号池，但两类单据在本地数据中共用同一个 ERP 凭证 ID 字段；ERP 凭证号本身全局唯一，不存在该碰撞问题。

问题发现：

1. 在主动走读撤核销定时任务代码时，发现旧逻辑只使用 ERP 凭证 ID 识别需要撤核销的数据，没有区分凭证来自收据接口还是发票接口。
2. 该设计等价于假设 ERP 凭证 ID 全局唯一，但真实约束是凭证 ID 只在各自接口命名空间内唯一。
3. 早期发票数量远低于收据，碰撞风险较低；随着发票增速超过收据，两类单据出现相同凭证 ID 的可能性明显增加。
4. 一旦号码碰撞，原本针对收据侧流水的撤核销可能同时命中发票侧的对账辅账，导致额外生成冲销辅账、废弃收据流水并将发票订单重新放回发票未对账池。

测试验证：

1. 在测试环境构造两笔对账辅账：其中一笔的收据 ERP 凭证 ID 为 `X`，另一笔的发票 ERP 凭证 ID 也为 `X`。
2. 修改前，撤核销任务同时命中两笔对账辅账，两者都进入冲销、流水废弃和订单回到未对账池的处理链路。
3. 修改后，只有收据 ERP 凭证 ID 为 `X` 的对账辅账执行撤核销，发票 ERP 凭证 ID 为 `X` 的另一笔对账辅账不受影响。

修复方案：

1. 将撤核销数据的识别条件从单一 ERP 凭证 ID 改为“ERP 凭证 ID + 接口名”组合业务键。
2. 收据使用 `ERP-RECEIPT`，发票使用 `ERP-INVOICE`，使相同凭证 ID 在不同接口命名空间中能被正确区分。
3. 完成问题上报、测试复现、修复方案设计、补丁开发和生产上线。

生产验证：

1. 补丁上线约一个月后，上游发生流水重推，需要对已核销的原批次流水执行撤核销。
2. 当时对账数据中已经存在与原批次收据 ERP 凭证 ID 相同的发票 ERP 凭证 ID。
3. 撤核销定时任务执行后，收据侧对账辅账正常撤核销，相同 ERP 凭证 ID 的发票侧对账辅账没有受到影响，生产场景验证了组合业务键修复的有效性。

### 4.6 未对账池批量终止与恢复

业务背景：

1. 流水以收据辅账形式通过 `ERP-RECEIPT` 入账，订单以发票辅账形式通过 `ERP-INVOICE` 入账。
2. ERP 入账成功并回写凭证信息后，定时任务从辅账沿生成关系找到对应流水或订单解析明细，经过处理后生成收据未对账池或发票未对账池数据。
3. 2B / 2C 新场景如果因代码逻辑问题产生金额错误，错误侧辅账会在推送 ERP 时被拦截；另一侧辅账可能已经入账并进入未对账池，因缺少匹配数据而无法在本地完成核销。
4. 财务会在 ERP 中手工录入正确的另一侧凭证，并与已经入账的收据或发票完成手工核销；本地系统需要把无法继续自动核销的单边数据从后续对账链路中受控退出。

功能设计：

1. 提供按钮和接口，支持批量提交 ERP 凭证号并填写终止原因备注。
2. ERP 凭证号对收据和发票全局唯一，因此无需额外指定单据类型；同一能力可以处理收据或发票单边入账形成的异常数据。
3. 根据凭证号定位未对账池中的核销凭证明细，将对账状态从“待对账”更新为“终止对账”，不删除原始业务数据，也不修改 ERP 已入账事实。
4. 每一笔被修改的核销凭证明细都会更新备注、更新人和更新时间，保留人工处理原因与审计信息。

状态约束：

1. 允许“待对账 -> 终止对账”。
2. 禁止“对账成功 -> 终止对账”，避免已经完成核销的数据被错误作废。
3. 重复提交时过滤已经处于“终止对账”的记录，避免重复处理。
4. 支持“终止对账 -> 待对账”的逆向恢复，用于纠正人工误操作。

实际使用：

1. 该能力上线后长期用于金额计算错误、特殊业务场景等异常数据收口。
2. 平均每个自然月 / 账期使用约 5 次，每次处理几十笔辅账。
3. 该功能将临时数据修复转化为可重复使用、可审计、可恢复的运营能力，同时保持本地核销状态与 ERP 手工账务处理的职责边界。

### 4.7 对账池生成任务的动态分区范围优化

业务变化：

1. 系统早期尚未接入券包业务，2C 支付订单通常在当月生成辅账并推送 ERP 入账。
2. 辅账入账成功后，对账池生成定时任务捞取“已入账、未进入对账池”的辅账，再沿生成关系映射逐层追溯到订单解析明细，生成发票未对账池数据。
3. 券包业务接入后，一年期券包只有在完全到期或使用完毕后才能生成辅账，因此当月入账的辅账可能需要追溯一年甚至更早的订单明细。
4. 2C 各业务场景共用辅账表、关系映射表、订单解析明细表和同一套对账池生成管道。

原方案问题：

1. 定时任务通过配置固定回溯前 `X` 个月的数据；为兼容一年期券包，配置需要从原来的近三个月扩大到十二个月。
2. 十二个月仍不能完整覆盖辅账延期到十三、十四个月的场景，继续扩大窗口也无法从机制上保证完整性。
3. 普通当月订单为了兼容少量跨期券包，也需要扩大到跨年度查询范围，增加物理分区访问范围和分区内索引检索成本。
4. 订单解析明细表一年可以承载数亿笔数据，但由于查询使用索引，不能表述为每次实际全量扫描数亿行；真实性能问题来自扩大分区和索引查询范围。

隐藏缺陷：

1. 如果固定窗口内完全找不到订单明细，旧逻辑不会更新辅账的对账状态，后续仍有机会重新处理。
2. 如果固定窗口只命中同一辅账下的部分订单明细，旧逻辑会继续生成未对账池数据，并将辅账更新为“已入对账池”。
3. 这种部分成功状态会导致窗口外的订单明细漏入对账池，而辅账又因为状态已完成而不再被后续任务重新处理。

MySQL 分区背景：

1. 订单解析明细表按 `business_day` 做周分区，关系映射表按创建时间做月分区。
2. 系统中其他大表还存在按月或每三天分区的方式，具体分区粒度根据业务数据量和访问模式确定。
3. 未来时间分区需要提前在 DDL 中创建。
4. 当前记忆中的订单明细表结构接近 `PRIMARY KEY (id, business_day)` 与 `PARTITION BY RANGE COLUMNS (business_day)`；准确 DDL 仍以原系统为准，不在简历中展开。

优化方案：

1. 利用“同一辅账只聚合同一交易日期订单”的业务约束，将订单交易日期作为生成链路的最早时间。
2. 以辅账创建时间作为沿关系映射向前追溯的最晚时间。
3. 每次处理 200 笔辅账，提取这一批数据涉及的分区字段，使用最小交易日期和最大辅账创建时间动态计算查询范围。
4. 查询关系映射和订单解析明细时显式带入对应时间范围，使 MySQL 可以缩小需要访问的物理分区，并收紧分区内的索引检索范围。
5. 移除对固定前 `X` 个月配置的依赖，使普通当月订单使用较小查询范围，跨期券包订单则按真实业务时间跨度扩大范围。

批次大小取舍：

1. 每批 200 笔是系统原有配置，也是公司文档推荐值。
2. 测试阶段比较过每批 100、200、500 笔，三种配置没有表现出明显的性能差异。
3. 综合内存占用、系统稳定性和已有运行经验，继续沿用每批 200 笔。
4. 改造后的整体任务性能已经满足业务规定基线，因此没有为了追求局部测试中的微小差异继续扩大批次。

结果：

1. 对账池生成定时任务由接近 40 分钟缩短到约 8 分钟。
2. 动态范围能够覆盖超过十二个月的跨期订单，不再需要不断扩大固定月份配置。
3. 同一辅账对应的订单明细能够在完整时间范围内统一处理，规避部分命中后错误标记“已入对账池”导致的订单漏入池问题。
4. 测试阶段完成数据完整性与状态处理验证；上线后与账务用户持续协作观察，任务连续运行三个月未发现运行问题。

### 4.8 亿级流水表的分页查询与索引优化

问题背景：

1. 流水表整表数据量达到亿级。
2. 系统使用 MyBatis 体系内的分页组件，输入 `pageNum` 和 `pageSize` 后执行分页明细 SQL 与 `COUNT(*)`，并返回分页数据、总数等信息；当前记忆更接近 PageHelper，但尚未完全确认，简历中不直接写具体插件名称。
3. 总数统计在部分条件下需要约 10 秒，原同步链路会导致分页列表必须等待 `COUNT(*)` 完成后才能整体返回。

接口与深分页优化：

1. 将分页数据与总数统计拆分为两个独立接口，由前端分别请求、分别展示，使用户可以先看到列表数据，不再被总数统计阻塞。
2. 分页数据先根据过滤条件和排序查询 row ID，再按 row ID 获取完整流水记录，减少深分页过程中无效回表的成本。
3. SQL 分页语义为 `LIMIT offset, page_size`，其中 `offset = (pageNum - 1) * pageSize`；准确 SQL 结构以原系统为准。

隐藏性能缺陷：

1. 大部分分页查询可以在 3 秒以内返回，但当 `LIMIT 20` 而实际结果不足 20 条时，查询反而经常超过系统设置的 120 秒超时。
2. 结果能够较快填满一页时，MySQL 找到满足 `LIMIT` 的数据后可以提前结束；结果不足一页时，需要继续搜索以确认没有更多记录，暴露了原访问路径缺少合适索引的问题。
3. 通过 `EXPLAIN` 检查执行计划，发现异常查询的访问类型退化为 `ALL`。

索引修复：

1. 典型过滤条件包括分片号、处理状态和批次号，并按 `create_time` 排序。
2. 原联合索引覆盖分片号、处理状态和批次号，但没有包含同时承担分区与排序作用的 `create_time`。
3. 将索引调整为接近“分片号、处理状态、批次号、`create_time`”的顺序，使前三项等值过滤后可以继续利用索引中的时间字段完成范围与顺序访问。
4. `create_time` 同时是该表的时间分区字段；优化收益来自缩小物理分区访问范围、利用联合索引顺序和降低深分页回表成本，不能泛化为所有 `ORDER BY` 字段都必须加入索引。

结果：

1. 结果不足 20 条的尾页查询从经常超过 120 秒超时降到稳定 3 秒以内。
2. 主要分页查询稳定在 3 秒以内。
3. `COUNT(*)` 仍可能需要约 10 秒，但由于已经拆分为独立请求，不再阻塞分页列表展示。

### 4.9 对账池生成任务的中断续跑

原方案问题：

1. 对账池生成任务先捞取辅账，再沿映射关系分批查询明细并写入未对账池。
2. 旧方案只有辅账级完成状态，必须等全部明细入池后才将辅账更新为“已入对账池”。
3. 如果服务器在明细只处理一部分时宕机，已经入池的数据会保留，但辅账仍处于未完成状态；下次从整笔辅账重新执行时，已成功部分可能重复入池。
4. 对账池没有数据库唯一键。业务唯一键过长，增加唯一索引会带来索引空间、写入维护和批量插入性能成本，因此重复控制主要依赖任务互斥、处理状态和事务边界。

任务调度与并发控制：

1. 定时任务采用抢占式调度，数据库通过“状态为待执行时才能更新为执行中”的条件更新限制同名任务启动。
2. 抢占成功后设置 Redis 分布式锁，锁 value 保存本次执行的唯一标识。
3. Redis 锁 TTL 为 6 小时，与公司“单个定时任务不得超过 6 小时”的运行基线一致。
4. 任务在 `finally` 中释放锁时会校验锁 value，只允许当前执行实例释放自己持有的锁。
5. 数据本身带有分片号，由任务按分配的分片范围处理。

明细级 Checkpoint：

1. 为源明细增加“未入对账池、已入对账池、入对账池失败”等处理状态。
2. 每批明细使用一个本地事务，将“批量写入未对账池”和“更新源明细为已入对账池”一起提交。
3. 如果提交前宕机，两部分数据一起回滚；如果提交后宕机，入池数据和成功状态同时保留。
4. 后续任务只查询“未入对账池 / 入对账池失败”的明细，跳过已经成功的数据。
5. 全部明细处理结束后再次扫描状态，只有不存在未入池或失败明细时，才将辅账更新为“已入对账池”。

异常恢复：

1. 服务器直接宕机时，Redis 锁最终会因 TTL 到期而释放，但数据库任务状态仍保持“执行中”，因此新的同名任务不能自动抢占。
2. 系统提供人工强制恢复能力：在确认旧任务已经停止后，删除选中任务的 Redis 锁，并将数据库中的旧任务状态重置为停止。
3. 定时任务支持手动启动；重新启动后根据明细状态从未处理或失败的数据继续执行。
4. 强制恢复属于运维兜底，执行前必须确认旧实例不再运行，避免两个实例同时处理相同数据。

验证结果：

1. 测试环境主动中断任务后重新启动，已成功批次没有重复写入，未完成批次能够继续执行。
2. 生产服务器重启场景下，会先确认相关定时任务均已停止，再重新启动任务，能够按照明细状态继续处理。

### 4.10 对账池生成任务 OOM 与 MyBatis 缓存

问题现象：

1. 对账池生成任务曾在生产环境发生 OOM，所在服务器物理内存为 16GB。
2. 分析 Heap Dump 后，发现 `ReconcileData` 类及其关联对象占用或保留内存超过 10GB；`ReconcileData` 是生成后需要写入对账池的数据对象。
3. 对象引用链指向 MyBatis 缓存组件，说明业务处理结束后，大批量查询结果仍被缓存引用，无法被 GC 及时回收。

根因判断：

1. 可以确认的问题是：MyBatis 缓存生命周期与大批量、一次性查询场景不匹配，持续持有包含 `ReconcileData` 的查询结果集合，导致任务运行期间内存不断上升。
2. 早期没有发生 OOM，不能直接解释为对象当时能够正常释放；更可能是早期数据量、查询次数、单次结果大小或并发量尚未超过 JVM 堆的临界点，随着业务数据增长才暴露问题。
3. 当前记忆是通过注解关闭了对应 Mapper 查询方法的缓存，形式可能接近 `@Options(useCache = false)`，但已经无法恢复原代码确认具体注解和配置。
4. 当前无法确认查询与写入是否属于同一个 Mapper namespace，也无法进一步断言是一级缓存、二级缓存、事务缓存缓冲区或自研框架封装中的哪一层。

修复与验证：

1. 关闭对应大批量查询方法的 MyBatis 缓存，避免一次性结果集合在业务处理完成后继续被缓存持有。
2. 在测试环境主动调低服务可用内存，使用相同任务进行缓存开关对照：保留缓存时有较高概率触发 OOM，关闭缓存后能够完成任务。
3. 修复上线后连续运行三个月，未再次发生 OOM。

事实边界：

1. Heap Dump 中的 10GB 可能是对象及其关联对象的保留内存，不能在没有原始分析报告的情况下断言全部是 `ReconcileData` 对象自身的浅堆内存。
2. 不将该问题直接表述为经典内存泄漏，更准确的说法是“缓存长期持有大批量查询结果，造成对象滞留和内存持续增长”。
3. 不对已经无法确认的 Mapper 结构、缓存层级、事务范围和历史配置变化补充推测性结论。

## 5. 边界和风险

1. Redis 定时任务锁、下游令牌限流和按年份分库属于参与或了解范围，不能表述为本人完整设计。
2. 支付流水号索引原本已经存在，性能案例应强调 N+1 定位和批量查询改造。
3. 延长数据库锁等待时间只能作为缓解措施，不能替代正确事务边界和锁竞争治理。
4. 冲销链路依赖重试、ERP 幂等和状态推进，属于跨系统最终一致性，不表述为本地事务或分布式事务一次性完成。

## 6. 面试表达入口

### 6.1 项目定位

> 我主要负责云服务结算系统中的交易流水处理与核销业务域。上游是 FFP 账单文件，下游是 ERP，核心链路包括文件解析入库、流水批次处理、业务规则和订单匹配、辅账生成、ERP 推送回写，以及流水和订单的核销对账。这个业务域的难点不是单个接口，而是保证长链路不重不漏、状态一致、失败可恢复，并在百万级数据量下维持批处理性能。

### 6.2 SQL 与批处理优化入口

> 一次流水匹配订单任务每分钟不足 1000 笔，但慢 SQL 日志里没有超过 1 秒的查询。我检查代码后发现是循环逐笔查询造成的 N+1 数据库访问。之后把流水按 1000 笔批量查询订单，并按支付流水号构建 Map 做内存匹配，单批查询控制在 1 秒以内，整体达到 100 万笔 8 分钟以内的处理基线。

### 6.3 生产事故入口

> 一次用户反馈对账辅账推送时原收据余额不足。我们从批次和日志定位到重复、漏生成并存，影响数千笔辅账和百万级关联明细。日志存在锁等待超时和 JDBC 回滚异常，代码中又存在事务方法同类自调用。我的修复是把状态更新和辅账生成迁移到独立 Spring Service，重新建立真正经过代理的事务边界；测试验证一致性后上线，同类问题未再发生。

### 6.4 OOM 排障入口

> 对账池生成任务曾发生 OOM。我通过 Heap Dump 发现 `ReconcileData` 及其关联对象占用或保留内存超过 10GB，引用链指向 MyBatis 缓存组件。结合任务特点，判断是大批量查询结果被缓存长期持有，导致业务处理完成后对象仍无法及时回收。之后关闭对应查询缓存，并在调低服务内存的测试环境中做缓存开关对照验证；修复上线后连续运行三个月，未再次发生 OOM。由于已经无法恢复原代码，我不会进一步声称具体是一、二级缓存或某个 Mapper namespace 的刷新问题。
