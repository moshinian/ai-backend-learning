# SQL 性能分析与批量访问优化

## 1. 本质

SQL 性能优化不是看到慢就加索引，而是先判断耗时来自哪里：单条 SQL 执行慢、SQL 调用次数过多、扫描范围扩大、锁等待、连接池等待，还是应用和下游处理缓慢。

分析单位既包括一条 SQL，也包括完整业务任务的数据访问模型。

## 2. 解决的问题

1. 单条查询扫描和返回过多数据。
2. 索引与查询条件不匹配。
3. N+1 查询造成大量数据库往返。
4. 深分页、排序、分组和临时结果成本过高。
5. 长事务和大范围更新造成锁等待。
6. 数据长期增长后，原有任务和访问路径不再满足吞吐要求。

## 3. 定位机制

标准顺序：

```text
确认业务现象和数据规模
-> 统计单次耗时、执行次数、扫描量和返回量
-> 查看慢日志、监控和调用链
-> 使用 EXPLAIN 分析访问路径
-> 针对性修改
-> 用相同数据规模验证结果
```

`EXPLAIN` 重点关注：

1. `type`：访问方式，`ALL` 需要结合表大小和实际成本判断。
2. `key`：优化器实际采用的索引。
3. `rows`：预计扫描行数。
4. `filtered`：条件过滤后的预计比例。
5. `Extra`：覆盖索引、额外排序、临时表、索引条件下推等信息。

慢日志里没有记录，不等于数据库访问没有性能问题。大量不足慢日志阈值的小查询，累计耗时仍然可能很高。

## 4. 常见优化方向

### 4.1 索引与访问路径

1. 为高频 `WHERE`、`JOIN`、排序和分组设计合适索引。
2. 联合索引要结合最左前缀、等值条件、范围条件和排序需求。
3. 只查询需要的列，必要时评估覆盖索引。
4. 避免对索引列进行不必要的函数计算和隐式类型转换。
5. 索引不是越多越好，额外索引会增加写入和维护成本。

### 4.2 SQL 与调用方式

1. 避免 `SELECT *`。
2. 把逐笔查询改成受控大小的批量查询。
3. 不需要去重时使用 `UNION ALL`，但不能改变业务语义。
4. 大数据分页优先考虑基于稳定排序键的游标翻页，而不是无限增大的 offset。
5. 批量参数不能无限增大，需要平衡 SQL 解析、网络包、结果集内存和失败重试成本。

### 4.3 数据规模与事务

1. 大任务按时间或业务分片缩小单次查询范围。
2. 物理分区表查询需要携带有效分区条件，才能利用分区裁剪。
3. 已完成历史数据可以根据业务状态做冷热分离。
4. 更新条件应命中合适索引，减少扫描和锁定范围。
5. 缩短事务范围，避免在数据库事务内执行长时间远程调用。

## 5. 边界

1. `Using filesort`、`Using temporary` 或 `ALL` 不是脱离数据规模就能判定的错误。
2. 单条 SQL 优化后仍要检查整体调用次数和网络往返。
3. 缓存、分区、读写分离和分库属于更高层手段，不能替代错误 SQL 与访问模型的修复。
4. 批次大小不是固定答案，需要通过数据库响应、应用内存、下游能力和业务时限共同验证。

## 6. 项目或工程关联

流水匹配支付订单任务曾在几十万待处理数据下每分钟不足 1000 笔。慢日志阈值为 1 秒，但没有对应慢 SQL。代码检查发现程序在循环中为每笔流水查询一次订单，支付流水号索引虽然已经存在，大量数据库往返仍形成 N+1 问题。

优化方式：

1. 每 1000 笔流水提取一批支付流水号。
2. 使用批量条件查询订单，单批耗时不足 1 秒。
3. 将结果转换成支付流水号到订单的 Map。
4. 在内存中完成流水和订单匹配。

最终达到 100 万笔 8 分钟以内的项目处理基线。另一个长期增长问题通过按业务月份拆分核销任务和查询范围缓解；由于物理分区细节尚未确认，项目表达不把它夸大为本人设计数据库分区。

## 7. 面试表达入口

> SQL 优化要先区分单条 SQL 慢、调用次数过多、扫描范围大和锁等待。我会结合监控、慢日志、调用链和 EXPLAIN 检查实际索引、扫描行数和排序，再从联合索引、覆盖索引、SQL 改写、批量查询、分页和数据分片处理。我的实际案例不是新增索引，而是发现循环逐笔查询造成 N+1，改成每 1000 笔批量查询并用 Map 匹配，使百万笔数据达到 8 分钟以内。
