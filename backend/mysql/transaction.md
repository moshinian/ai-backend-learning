## 5. 事务解决什么真实工程问题

事务解决的不是“让 SQL 更规范”，而是：

> 多个数据库操作必须共同形成一个合法业务状态，不能只成功一部分。

例如账单解析入库时，通常会涉及：

1. 插入批次表
2. 批量插入明细表
3. 更新批次状态
4. 写处理日志
5. 生成后续结算任务或标记待处理数据

如果没有事务，可能出现批次记录存在、明细只插入一半、状态却已经是成功。后续结算任务如果基于这个状态继续执行，就会导致漏结、重复结算或生成错误辅账。

所以本地事务适合保护：

| 场景 | 是否适合本地事务 |
|---|---|
| 同一个数据库内的批次、明细、状态更新 | 适合 |
| 更新订单状态和写本地操作日志 | 适合 |
| ERP 推送和本地状态更新 | 不能只靠本地事务 |
| MQ 发送和数据库提交 | 需要额外可靠性设计 |

更准确的项目表达：

> 本地数据库内部的状态变更用事务保证原子性；跨系统调用不依赖一个大事务，而是通过状态机、幂等键、重试补偿和对账保证业务最终一致。

## 6. ACID 的工程含义

| 特性 | 解决的问题 | 人话理解 |
|---|---|---|
| Atomicity 原子性 | 中途失败导致部分 SQL 生效 | 要么全做，要么全不做 |
| Consistency 一致性 | 事务前后违反数据库约束或业务规则 | 事务执行后仍是合法状态 |
| Isolation 隔离性 | 并发事务互相干扰 | 不能让别的事务中间态影响当前事务判断 |
| Durability 持久性 | 提交成功后宕机导致数据丢失 | 已提交事务必须能恢复 |

注意：

ACID 里的“一致性”不是分布式系统里的最终一致性，也不是主从复制一致性。它更强调事务前后数据仍满足约束和业务规则。

## 7. 并发事务的三类读问题

| 问题 | 本质 | 例子 |
|---|---|---|
| 脏读 | 读到其他事务未提交的数据 | 读到临时的 `SETTLED`，对方回滚后该状态从未成立 |
| 不可重复读 | 同一事务两次读同一行，内容变了 | 第一次读订单是 `INIT`，第二次变成 `SETTLED` |
| 幻读 | 同一事务两次范围查询，结果集变了 | 第一次查 10 条未结算订单，第二次查到 11 条 |

区别：

1. 脏读关注“读到了未提交数据”。
2. 不可重复读关注“同一行内容变化”。
3. 幻读关注“范围查询结果集变化”。

## 8. 隔离级别

隔离级别的本质是：

> 在一致性和并发性能之间做取舍。

| 隔离级别 | 能避免什么 | 仍可能有什么问题 |
|---|---|---|
| Read Uncommitted | 基本不避免 | 脏读、不可重复读、幻读 |
| Read Committed | 脏读 | 不可重复读、幻读 |
| Repeatable Read | 脏读、不可重复读 | 标准 SQL 下仍可能有幻读 |
| Serializable | 基本都避免 | 并发最低，性能代价最大 |

MySQL InnoDB 默认隔离级别是 `Repeatable Read`。

不能简单说 MySQL RR “无法处理幻读”。更准确是：

1. 普通 `select` 属于快照读，RR 下通过 MVCC 读取一致性视图，通常不会看到新插入的幻影行。
2. `select ... for update`、`update`、`delete` 属于当前读，需要通过行锁、间隙锁、Next-Key Lock 控制范围插入。

## 9. MVCC 的本质

MVCC 的本质：

> 不让普通读都加锁，而是通过多版本和可见性规则，让事务读取自己应该看到的版本。

核心组件：

| 组件 | 作用 |
|---|---|
| 隐藏事务 ID | 标记某个行版本由哪个事务生成 |
| undo log | 保存旧版本，支持回滚和历史版本读取 |
| Read View | 判断某个版本对当前事务是否可见 |

流程理解：

```text
事务 A 开始，生成 Read View
事务 B 把订单 status 从 INIT 改成 SETTLED 并提交
事务 A 再次普通 select
发现最新版本 SETTLED 对自己不可见
沿 undo log 找到旧版本 INIT
返回 INIT
```

所以 MySQL RR 仍有较好并发性能，是因为普通读不阻塞写，写也不阻塞普通读。

## 10. undo log、redo log、binlog

| 日志 | 所属层级 | 解决的问题 |
|---|---|---|
| undo log | InnoDB | 回滚、MVCC 历史版本读取 |
| redo log | InnoDB | 崩溃恢复，保证持久性 |
| binlog | MySQL Server | 主从复制、时间点恢复、变更订阅 |

容易混淆点：

1. `undo log` 不负责事务提交后的崩溃恢复。
2. `redo log` 才主要保证事务提交后的持久性。
3. `binlog` 是复制和恢复的基础，但不等于天然保证主从强一致，仍可能有复制延迟。

redo log 和 binlog 需要两阶段提交，是为了保证 InnoDB 事务状态和 Server 层 binlog 记录一致：

```text
1. redo log prepare
2. 写 binlog
3. redo log commit
```

如果只写 redo log 成功而 binlog 失败，主库可能能恢复出已提交事务，但从库和基于 binlog 的恢复看不到这次变更，导致主从不一致和恢复缺失。
