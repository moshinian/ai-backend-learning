# Java Map 与 ConcurrentHashMap

## 1. 本质

`HashMap`、`ConcurrentHashMap` 和 `Hashtable` 都提供 key-value 映射，主要区别是并发语义、空值约束和实现方式。

1. `HashMap` 非线程安全，允许一个 null key 和多个 null value。
2. `Hashtable` 是早期同步容器，不允许 null key/value，主要方法竞争同一个对象监视器。
3. `ConcurrentHashMap` 面向共享并发读写，不允许 null key/value，通过无锁读、CAS、桶级同步、协作扩容和分散计数降低竞争。

## 2. 解决的问题

`ConcurrentHashMap` 要解决的是：在保证单次 Map 操作并发安全和可见性的同时，避免所有读写操作竞争一把全局锁。

它不是完全无锁，而是根据操作复杂度选择不同控制方式。

## 3. Java 8 以后主要机制

一次 `put` 可以概括为：

```text
计算 hash 和桶下标
-> 桶为空：CAS 写入首节点
-> 桶正在迁移：参与或协助扩容
-> 桶不为空：锁当前桶头，在链表或红黑树中插入 / 更新
-> 更新分散计数，必要时触发扩容
```

### 3.1 空桶使用 CAS

多个线程同时看到同一桶为空时，通过“比较当前位置是否仍为 null，再原子写入新节点”竞争。只有一个线程成功，失败线程重新读取并重试，不会覆盖成功线程的数据。

### 3.2 冲突桶使用局部 `synchronized`

桶内插入涉及查重、遍历和链接节点等多步骤操作，需要互斥。实现会锁住当前桶头节点，并在加锁后再次确认它仍是该桶的头节点。

不同桶可以并发更新；只有落到同一桶的写操作互相竞争。这就是“桶级同步”，不是整张表的一把锁。

### 3.3 读取通常不加互斥锁

`get` 主要进行 hash 定位和链表或红黑树遍历。table 访问以及节点的 value、next 等字段通过原子访问和可见性语义保证已完成更新能被后续读取观察到。

无锁读取不等于获取整张 Map 的统一快照。并发修改期间，`size()`、遍历等聚合结果可能反映变化中的状态。

### 3.4 协作扩容

扩容时通过控制字段和迁移下标把旧表桶区间分配给多个线程。已迁移的旧桶放置转发节点，后续访问可以转向新表；其他线程遇到迁移状态时可以协助搬迁，而不是只能等待单个线程完成全部扩容。

### 3.5 分散计数

高竞争时，元素数量不会只依赖所有线程争用一个计数变量，而会分散到多个计数单元后汇总，思路类似 `LongAdder`。

### 3.6 冲突树化

桶内冲突达到条件后可以由链表转为红黑树，改善严重哈希冲突时的查找复杂度。但同一桶的写操作仍然会竞争，因此 key 的 `hashCode()` 质量仍然重要。

## 4. 边界

1. Java 7 的 `ConcurrentHashMap` 常用 Segment 分段锁描述；Java 8 以后不能继续把 Segment 当作主要实现。
2. 线程安全容器不等于多步业务逻辑自动原子化。

```java
if (!map.containsKey(key)) {
    map.put(key, value);
}
```

以上仍存在竞态，应根据语义使用 `putIfAbsent`、`computeIfAbsent`、`compute` 等原子复合操作。

3. 如果 Map 初始化完成后只读，并且被安全发布，普通 `HashMap` 可以满足要求；只有共享并发修改时才需要并发容器。
4. `computeIfAbsent` 等计算函数不宜执行长时间数据库或远程调用，否则可能放大相关 key 或桶的等待。

## 5. 项目或工程关联

交易流水规则系统存在“规则 ID -> Spring Bean 实现”的策略注册表。若该 Map 只在启动阶段构建并在运行期只读，可以使用安全发布的普通 Map；若运行期支持动态新增或替换规则，则需要并发 Map 和原子更新方法。

当前只能确认规则实现由 Spring 管理并进入 Map，尚未确认注册表的具体构建和动态更新方式，因此项目表达不主动声称使用了 `ConcurrentHashMap`。

## 6. 面试表达入口

> Java 8 以后的 ConcurrentHashMap 不是完全无锁，而是读操作通常不加互斥锁，空桶首节点用 CAS 插入，发生冲突时只锁当前桶，扩容时多个线程可以协作迁移，计数也会分散竞争。它保证容器提供的原子操作，但 containsKey 再 put 这种多步组合仍需改成 putIfAbsent 等原子方法。
