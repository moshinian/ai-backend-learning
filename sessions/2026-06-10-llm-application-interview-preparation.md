# 2026-06-10 大模型应用开发工程师面试准备

## 1. 背景

本次面试是临时加入的机会，岗位重点包括：

1. Multi-Agent 系统设计与任务协作
2. RAG、企业知识库和检索优化
3. Dify、LangGraph、Spring AI、MCP、A2A
4. Docker、K8s、大模型部署和高并发
5. Kafka、Flink、Hive 等实时计算组件
6. 物流业务理解和技术方案转化

这份 JD 覆盖范围很广。当前不因为单次临时面试全面改变学习路线，先用它验证市场对“Java 后端 + RAG 应用”转型方向的要求。

---

## 2. 当前匹配判断

### 已有优势

1. Java 后端和企业业务系统开发经验
2. 结算系统中的状态机、MQ、幂等、补偿、批处理和多节点并行经验
3. 从零搭建 RAG 知识问答系统的经历
4. 能从业务问题出发设计工程方案

### 当前缺口

1. 缺少 Multi-Agent 生产实践
2. Dify、LangGraph、Spring AI、MCP、A2A 尚未形成实战闭环
3. Flink、Hive、K8s 和推理优化不是当前强项
4. RAG 项目此前缺少正式的评测体系、Rerank 和生产指标表达

面试原则：

1. 不虚构生产经验
2. 讲清已有后端能力如何迁移到 Agent 系统
3. 区分“实际做过”“理解原理”“后续优化方向”
4. 不把扩大 `TopK` 描述成准确率问题已经解决

---

## 3. 本次已经完成的准备

1. 修正 1 分钟自我介绍
2. 整理 2 分钟 RAG 项目介绍
3. 区分关键词检索与向量检索
4. 理解加权融合、RRF 和 Rerank 的作用
5. 区分 `Recall@K`、MRR、NDCG
6. 从检索、生成、工程三个层面评价 RAG
7. 分析整篇向量化的问题
8. 分析 Chunk 大小、Overlap 和语义边界
9. 理解 Parent-Child Retrieval 的目标和工作方式
10. 分析“正确 Chunk 已进入 TopK，但模型仍回答错误”的原因

---

## 4. 已暴露问题

1. 自我介绍容易用“我觉得可以胜任”代替事实证据。
2. RAG 项目介绍容易堆功能，没有按问题、架构、职责、难点和指标组织。
3. 一开始把 `Top3` 改为 `Top5` 当成准确率问题的解决方案。
4. 对检索结果融合的初始表达是“调整权重取最小误差”，但没有先定义标注集和评测指标。
5. 评价 RAG 时容易混合检索质量、生成质量和系统容量指标。
6. Chunk 参数容易直接给固定值，没有说明单位、文档结构和评测依据。
7. 对 Parent-Child Retrieval 没有已有认知，目前只作为后续优化方向，不应表达为已实现功能。

---

## 5. 尚未完成

1. Multi-Agent 物流场景系统设计口述
2. 结算系统并发安全、幂等和失败恢复的 2 分钟回答
3. MCP、A2A、LangGraph、Spring AI 的口述验证
4. 没有相关生产经验时的压力问答
5. Docker、K8s、Kafka、Flink、Hive 的岗位针对性复盘

---

## 6. 面试后的恢复入口

面试结束后优先记录：

1. 面试官实际问题
2. 当时的真实回答
3. 哪些问题回答顺畅
4. 哪些问题完全不会或表达失真
5. 面试官对项目、技术栈和经历的反馈
6. 是否需要调整后续学习任务池

只有当多次面试都稳定出现 Multi-Agent、RAG 工程化或大数据能力要求时，再正式修改长期 Roadmap。
