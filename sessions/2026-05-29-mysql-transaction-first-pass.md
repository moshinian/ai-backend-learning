# 2026-05-29 MySQL 事务第一轮

## 1. 本次学习内容

本次围绕 MySQL 事务主线做第一轮学习：

1. 事务为什么出现
2. ACID 分别解决什么工程问题
3. 脏读、不可重复读、幻读的区别
4. 隔离级别为什么是性能和一致性的折中
5. MySQL InnoDB 默认 Repeatable Read 的基本含义
6. MVCC、Read View、undo log 的关系
7. undo log、redo log、binlog 的职责边界
8. redo log 和 binlog 为什么需要两阶段提交

## 2. 已基本掌握

1. 事务用于保证同库内一组数据库操作整体成功或整体失败。
2. ACID 里的“一致性”不等于分布式最终一致性。
3. 脏读是读到未提交数据，不可重复读是同一行内容变化，幻读是范围结果集变化。
4. Serializable 不适合默认使用，因为会显著降低并发能力。
5. MVCC 不是复制查询结果，而是通过多版本和 Read View 判断行版本可见性。
6. 跨系统调用不能只靠本地事务，需要状态机、幂等、重试补偿和对账。

## 3. 暴露的问题

1. 解释脏读时容易混入并发写覆盖问题。
2. 一开始倾向于认为避免不可重复读就是禁止其他事务写，缺少 MVCC 视角。
3. 把 undo log 误认为负责事务提交后的持久性。
4. binlog 的作用容易说成“保证主从一致”，忽略复制延迟和两阶段提交边界。
5. “事务保证业务逻辑正确”这个表达过大，事务只能保证数据库操作边界。

## 4. 关键修正

1. `undo log` 负责回滚和 MVCC 历史版本。
2. `redo log` 负责崩溃恢复和持久性。
3. `binlog` 负责复制、恢复和订阅。
4. MySQL RR 下，普通 `select` 属于快照读，当前读需要结合锁理解。
5. 项目表达中要说“跨系统最终一致”，不要说“跨系统事务一致”。
