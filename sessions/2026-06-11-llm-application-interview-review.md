## 2. 表现判断

### RAG 架构

能够拆分知识库构建和在线问答两条链路，也能说明 Spring Boot、FastAPI、PostgreSQL 和 Redis 的职责。

主要问题：

1. 初始回答偏功能罗列，缺少业务目标、数据流和工程保障。
2. 把归一化加权融合称为 Rerank。
3. 尚未实现独立 Rerank，不能包装成现有能力。

### Chunk

能够说明固定窗口、Overlap、自然边界、文档类型差异和离线评测，工程意识较好。

需要改进：

1. 参数必须说明字符或 Token 单位。
2. 参数应表达为初始实验值，不能说“200 或 500 均可”。
3. 优先在最大窗口内向前寻找自然边界，避免无上限向后扩展。

### 混合检索与评测

现场把 Prompt、模型温度和模型选型混入混合检索优化，检索层和生成层边界不清。

复盘后已明确：

1. 全文检索和向量检索负责单路召回。
2. 归一化加权或 RRF 负责结果融合。
3. Cross-Encoder Rerank 负责候选集二阶段重排。
4. `Recall@K` 看是否召回，MRR 看第一个相关结果排名，NDCG 看分级相关性和整体排序。

### Prompt

能够想到离线评测和场景化 Prompt，但缺少模板结构、版本管理和回归机制。

需要稳定表达：

1. Prompt 只负责最终生成时不会改变检索排序。
2. 模板应拆分角色、规则、上下文、历史、问题和输出格式。
3. 场景化 Prompt 应通过意图分类路由，而不是为单个问题持续增加特例。

### Agent

现场回答偏向 Qwen 和 DeepSeek 的模型选型，没有回答 Agent 编排。

复盘后已明确：

1. 当前项目是固定 RAG Pipeline，不是 Agent。
2. Agent 的关键不是工具数量，而是动态决策权和状态闭环。
3. 没有生产经验时要先声明边界，再说明结算系统中的状态机、幂等、补偿和任务编排能力如何迁移。

### 结算系统

能够说明 2C、普通 2B、商推和流水四类业务，日订单规模约 600 万，并说明分区、冷热分离和按年份拆库的演进。

主要问题：

1. 业务场景、系统架构、技术栈和性能优化混在一起。
2. 多次把“技术栈”口误成“技术债”。
3. 从批处理耗时和数据库访问累计耗时直接推导“数据库是瓶颈”，证据链不完整。
4. 单条 SQL 不慢但累计访问耗时高，需要继续排查总扫描量、执行次数、重复查询、分页、锁等待和并行度。
