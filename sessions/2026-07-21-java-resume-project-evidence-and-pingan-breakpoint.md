## 本次做了什么

1. 确认求职策略为 Java 后端和 AI 应用两个岗位版本并行保留，不在简历文件职责上设置主次；实际投递以 Java 后端为主要方向，AI 应用机会继续保留。
2. 将原 AI 应用工程师简历归档到 `resume/ai-application-resume.md`，并按相同 Boss 直聘格式建立 `resume/java-backend-resume.md`。
3. 对 Java 后端简历做客观评估，确认主要竞争力来自荣耀平台、结算业务复杂度、流水和核销全链路经验；主要短板是项目证据、Java / Spring 机制深度和可追问的量化案例尚未充分写入。
4. 还原交易流水处理与核销业务域的完整链路，包括 FFP 文件集成、流水批次处理、场景与订单匹配、辅账生成、ERP 推送回写、收据和发票待对账池、核销及对账辅账。
5. 还原上游历史流水冲销管道：C/U 批次与原批次号关联、按处理阶段执行失效、负向冲销、撤核销和待对账恢复；三年内支撑不少于 50 次上游历史数据纠错。
6. 还原流水匹配订单性能案例：定位循环逐笔查询造成的 N+1 访问，改为每 1000 笔批量查询并构建 Map 匹配，达到 100 万笔 8 分钟以内的处理基线。
7. 还原对账辅账事故：从 ERP 原收据余额不足追踪到重复和漏生成，结合锁等待超时、JDBC 回滚异常和事务自调用问题，最终通过独立 Spring Service 重建事务边界，修复后未再复发。
8. 校准 `Could not roll back JDBC transaction` 的技术口径：它表示 Spring 调用 JDBC 回滚时发生异常，不能单独证明数据库已部分提交；真实数据结果仍需结合最底层异常、调用链和数据库状态确认。

## 关键结论

1. 结算系统主职责可以统一表述为“交易流水处理与核销业务域”，不需要勉强用单个功能名称概括。
2. 当前最强的三个 Java 项目证据是冲销补偿链路、百万级批处理优化和对账辅账事务事故。
3. 性能案例的贡献是识别 N+1 和改造数据访问模式；支付流水号索引原本存在，不能包装成本人新增索引。
4. 冲销管道是承接已有能力后完成场景适配、修复和长期维护，不能包装成从零设计。
5. 事务事故中，锁等待超时是触发条件，业务事务边界不可靠是数据不一致的关键问题，JDBC 回滚异常是当时连接层或数据库异常状态的补充证据。
6. 规则系统尚未完成代码事实还原，在确认配置、分支、接口、处理器和扩展方式之前，不给它强贴策略模式、工厂模式或责任链模式。
