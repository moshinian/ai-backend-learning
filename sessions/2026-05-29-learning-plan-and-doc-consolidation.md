## 4. 本次学习内容沉淀

### Tomcat 与线程模型

已经初步掌握：

1. Tomcat 能并发处理请求，不是因为一个请求创建一个新进程
2. 请求进入业务处理阶段时，通常会占用工作线程池中的一个线程
3. 慢请求拖垮 Web 服务，很多时候不是 CPU 打满，而是工作线程被同步阻塞
4. 调用数据库、下游 HTTP 服务、大模型等 IO 或远程结果等待，会长时间占用线程
5. 可用工作线程减少后，新请求会排队，最终响应时间变长、吞吐下降

当时仍需补齐：

1. Tomcat、Servlet、`DispatcherServlet` 三者的调用关系
2. Keep-Alive / 长连接与请求线程占用的关系
3. Servlet 容器和操作系统线程 / IO 的连接点

用户已经能讲清：

1. TCP 连接存在不等于 Tomcat 工作线程一直被占用
2. Poller 监听连接可读事件，有请求可读并进入业务处理时才占用工作线程
3. 同步慢请求会导致工作线程池打满、请求排队、最终超时

仍需复盘：

1. 异步化设计不能泛化成“再开个线程”，要明确业务线程池、MQ、SSE、轮询的适用场景
2. TCP 三次握手由操作系统内核协议栈完成，Tomcat 不直接处理每个握手包
3. socket 上传输的是字节流，不是 JVM 字节码
4. Filter 是 Servlet 规范，不是 Tomcat 私有方法
