# Transaction Mistakes

## 1. 当前主题

MySQL 事务、隔离级别、MVCC，以及它们和结算系统一致性设计的关系。

---

## 2. 记录范围

本文件只记录事务相关错误、误区和纠偏，不记录当前任务状态。任务状态、断点和下一步动作以 `LEARNING_BACKLOG.md` 为准。

重点记录：

1. 把事务和业务补偿混为一谈
2. 只会背 ACID，但说不出分别解决什么问题
3. 分不清脏读、不可重复读、幻读
4. 分不清隔离级别和锁 / MVCC 的关系
5. 分不清 undo log、redo log、binlog 的职责边界
6. 把单机数据库事务和分布式一致性混为一谈
7. 项目表达停留在“用了事务”，讲不清事务边界

---

### 错误 1：把脏读和并发写覆盖混在一起

错误倾向：

- 用余额扣减例子解释脏读时，同时讲了 A、B 两个事务互相覆盖写结果。

问题：

- 脏读的核心是“读到了其他事务未提交的数据”。并发写覆盖属于丢失更新或写冲突问题，不要在定义脏读时混进来，否则面试官会认为概念边界不清。

修正：

- 脏读：事务 A 读到事务 B 尚未提交的数据；如果 B 回滚，A 读到的数据从未真正成立。

项目表达：

- 批次状态被事务 B 临时改成 `PARSED_SUCCESS` 但未提交，事务 A 读到后触发结算；如果 B 最后回滚，A 就基于错误中间状态执行了后续流程。

### 错误 2：把“避免不可重复读”理解成只能禁止其他事务写

错误倾向：

- 认为 A 读取订单期间，不能允许 B 修改同一条订单。

问题：

- 这是锁的思路，但不是唯一方案。InnoDB 的普通快照读主要通过 MVCC 读取一致性视图，避免读写强阻塞。

修正：

- 目标是让同一事务内多次读取同一行时看到稳定视图。实现方式可以是锁，也可以是 MVCC。

项目表达：

- 结算场景不能只靠“读时禁止别人写”。更常见是状态条件更新、乐观锁、唯一约束、幂等键，以及必要时 `select ... for update`。

### 错误 3：把 undo log 说成负责持久性

错误倾向：

- 认为 undo log 负责事务提交后宕机恢复。

问题：

- 提交后宕机恢复、保证持久性主要靠 redo log。undo log 主要用于回滚和 MVCC 历史版本读取。

修正：

- `undo log`：回滚、MVCC 旧版本。
- `redo log`：崩溃恢复、持久性。
- `binlog`：主从复制、时间点恢复、变更订阅。

项目表达：

- 讲 MySQL 日志时要把“本机崩溃恢复”和“复制到从库或下游订阅”分开，不能混成一个日志能力。

### 错误 4：把 binlog 说成天然保证主从一致

错误倾向：

- 认为 binlog 存在就是保证主从一致。

问题：

- binlog 是主从复制和恢复的基础，但不等于强一致。仍然可能有复制延迟、从库回放失败、网络异常。redo log 和 binlog 之间还需要两阶段提交，保证主库事务状态和 binlog 记录一致。

修正：

- 更准确说法：binlog 用于复制、恢复和订阅；它支撑主从复制，但一致性还取决于两阶段提交、复制链路和读写策略。

项目表达：

- 如果结算状态刚写主库后立刻读从库，可能读到旧状态。关键状态查询应考虑走主库，或结合读写路由、延迟检测和补偿机制。

### 错误 5：把事务说成保证业务逻辑一定正确

错误倾向：

- 说事务的目的是保证当前业务逻辑执行结果正确。

问题：

- 事务只能保证数据库内一组操作按 ACID 规则执行，不能保证 SQL、状态机或业务规则本身设计正确。

修正：

- 事务保证的是数据库操作的原子性、隔离性和持久性等；业务正确性还需要约束、状态机、幂等、校验、补偿和测试共同保证。

项目表达：

- 结算系统里，本地事务保证批次、明细、状态更新不会部分成功；状态是否允许流转、是否重复结算，还要靠状态机、唯一约束和幂等设计。
