# Redis 分布式锁误区

## 1. 误以为加了 Redis 锁就能绝对只处理一次

错误理解：

> `process()` 外面加 Redis 锁，就基本能保证任务只处理一次。

纠正：

Redis 锁有过期、续期失败、误删和故障切换等边界。外部 ERP 调用成功但本地未写 DONE 时，锁也无法回滚外部副作用。任务可靠性要靠数据库状态机、`process_token`、租约、业务幂等键和补偿对账。

## 2. 误以为释放锁可以直接 DEL

错误理解：

> 当前 Worker 处理完任务后直接 `DEL lock_key`。

纠正：

锁可能已经过期并被其他实例重新获取。直接 `DEL` 会误删新锁。释放锁必须用 Lua 原子判断 `unique_value` 是否匹配，匹配时才删除。

## 3. 误以为看门狗续期能证明任务健康

错误理解：

> Redis 锁还在续期，说明业务任务还正常。

纠正：

看门狗只说明续期线程还在运行。业务线程可能死循环、卡外部调用或阻塞。还要有业务心跳、任务最大执行时间、超时回收和 `process_token`。

## 4. 误以为续期不用校验锁值

错误理解：

> 看门狗只需要定期 `EXPIRE lock_key`。

纠正：

旧锁过期后，新实例可能已获得同名锁。旧 Worker 如果不校验 value 就续期，会给新 Worker 的锁续期。续期必须校验 `unique_value`。

## 5. 混淆 process_token 和业务幂等键

错误理解：

> 直接把 `process_token` 作为 ERP 幂等键。

纠正：

`process_token` 标识当前这轮处理权，每次认领都会变化。ERP 幂等键应是稳定业务身份，例如结算单 ID 或 posting 请求 ID，并绑定请求内容 hash。

## 6. 把网络超时当作外部调用失败

错误理解：

> ERP 调用超时后，本地任务可以直接标记失败。

纠正：

超时只是调用方不知道结果。ERP 可能已经成功处理。应进入不确定状态，先查询外部结果，必要时用原幂等键重试，并通过对账兜底。
