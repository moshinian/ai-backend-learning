# RAG 项目面试表达

## 1. 项目背景

云服务结算系统涉及产品、开发和测试等多个角色，历史需求、接口和设计文档比较分散。团队成员很难快速定位需要的信息，造成重复沟通和人工检索成本。

因此搭建内部 RAG 知识问答系统，让团队可以通过自然语言查询历史资料，并获得带来源依据的回答。

---

## 2. 两分钟项目介绍

这个项目来源于团队内部的实际问题。云服务结算系统涉及产品、开发和测试等多个角色，历史需求、接口和设计文档比较分散，人员很难快速定位信息，导致重复沟通和人工查找成本较高。因此，我搭建了一套面向内部资料的 RAG 知识问答系统。

系统主要分为知识库构建和在线问答两条链路。知识库构建阶段负责文档上传、解析、清洗、Chunk 切分、Embedding 生成以及向量入库。在线问答阶段接收用户问题，通过 PostgreSQL 全文检索和 pgvector 语义检索召回相关 Chunk，再对两路结果进行归一化加权融合，选取候选内容组装 Prompt，调用大模型生成回答，同时保留来源引用和会话记录。

技术架构上，Spring Boot 负责知识库、文档任务、用户会话、状态管理和问答接口等业务能力；FastAPI 封装 Embedding 与 LLM 等模型服务；PostgreSQL 负责业务数据、全文检索和向量数据存储；Redis 用于缓存。我负责系统整体架构和核心链路的搭建。知识库构建任务采用状态落库、幂等和失败重试，避免长耗时任务因服务异常而丢失。

当前主要难点是检索质量。部分问题的相关 Chunk 无法进入 `TopK`，属于召回问题；另一些问题虽然召回了正确 Chunk，但排名不够靠前，属于排序问题。后续需要从 Chunk 切分、Embedding 模型、查询改写、元数据过滤和融合权重等方向优化，并考虑引入 Cross-Encoder Rerank 对少量候选进行二阶段重排。所有改动都需要通过标注问答集，对比 `Recall@K`、MRR、NDCG 和接口延迟验证。

---

## 3. 架构主线

```text
文档上传
  -> 解析与清洗
  -> Chunk 切分
  -> Embedding
  -> PostgreSQL / 向量存储
  -> 关键词与向量混合检索
  -> 归一化加权或 RRF 融合
  -> 可选 Cross-Encoder Rerank
  -> Prompt 组装
  -> LLM 生成
  -> 引用与会话记录
```

## 4. 高频追问

### 为什么同时使用关键词和向量检索？

关键词检索适合订单编号、接口名称、错误码和专有名词等字面特征明确的内容，但对同义表达和自然语言改写不敏感。向量检索适合召回语义相近但表述不同的内容，但对编号、缩写和错误码等精确标识区分不够稳定。

混合检索利用关键词检索保证精确匹配，利用向量检索扩大语义召回，再通过归一化加权或 RRF 融合两路候选。当前项目尚未接入独立 Rerank 模型，应将其表达为后续优化方向。

### 两种检索分数如何融合？

BM25 分数和向量相似度不在同一尺度，不能直接相加。可以先归一化后加权，也可以使用 RRF 按排名融合，再对候选结果执行 Rerank。

权重需要基于包含问题、相关 Chunk 和相关性等级的测试集，通过 `Recall@K`、MRR、NDCG 等指标调整，不能只根据个别问题凭感觉设置。

### Fusion 和 Rerank 有什么区别？

Fusion 用于合并全文检索和向量检索两路候选。归一化加权使用两路相关性分数，RRF 使用两路结果排名。

Rerank 位于 Fusion 之后。Cross-Encoder 将用户问题和每个候选 Chunk 联合编码，重新计算相关性分数，再选出最终 Chunk 进入 Prompt。它只能改善已召回候选的排序，不能找回第一阶段没有召回的内容。

### 当前系统为什么不是 Agent？

当前系统属于固定 RAG Pipeline：检索、融合、Prompt 组装、模型调用和会话保存的顺序由程序预先定义，LLM 只负责生成答案，不负责动态选择工具或决定下一步。

如果升级为 Agent，需要维护统一 State，让模型根据当前问题和工具结果决定是否调用工具、如何构造参数、是否重试或结束，形成受约束的“决策、执行、观察、再决策”闭环。

### 如何评价 RAG 效果？

1. 检索层：`Recall@K`、MRR、NDCG、证据完整性
2. 生成层：答案正确性、忠实度、引用准确性、完整性和拒答能力
3. 工程层：端到端延迟、首 Token 延迟、吞吐量、错误率、成本和可用性

需要把检索和生成分开评估，否则答案错误时无法判断问题发生在哪一层。

### 为什么不能整篇向量化？

整篇文档可能包含多个主题，向量会混合这些语义，使具体问题与文档整体向量的相似度下降。长文档还可能超过 Embedding 模型限制；召回后直接交给大模型也会增加 Token、延迟和上下文噪声。

### Chunk 如何切分？

优先按照标题、段落、列表、句子等自然边界切分，再通过最大 Token 数控制长度，并保留一定 Overlap，避免关键信息被边界截断。

参数不能固定。接口文档应尽量保证 URL、参数和返回值的关系不被破坏；规章制度需要保留章节层级；最终参数需要通过离线评测调整。

### Chunk 太大或太小有什么问题？

Chunk 太大会混合多个主题，降低检索定位精度，并增加 Token、延迟和上下文噪声。Chunk 太小会破坏事实和上下文完整性，例如把接口 URL、参数和返回值拆散，同时增加向量数量和检索碎片。

### 正确 Chunk 进入 TopK 后为什么仍回答错误？

可能原因包括：

1. 正确 Chunk 排名靠后
2. 片段证据不完整
3. 文档版本冲突
4. 上下文重复或组织混乱
5. Prompt 约束不足
6. 模型能力或生成参数不合适

诊断时先检查最终发送给模型的完整上下文是否足以支持答案，再判断是否属于模型问题。

### 什么是 Parent-Child Retrieval？

把文档同时切成较大的父 Chunk 和较小的子 Chunk。检索时使用语义集中的子 Chunk 提高定位精度，命中后通过 `parentId` 找回更完整的父级上下文，从而兼顾检索精度和证据完整性。

当前项目尚未实现该方案，应表达为针对 Chunk 信息被拆散问题的后续优化方向。

### Embedding 模型如何升级？

不同模型即使向量维度相同，也可能属于不同向量空间，不能将 `model-v1` 文档向量和 `model-v2` Query 向量直接混合检索。

任务和向量记录需要保存：

```text
model_name
model_version
vector_dimension
chunk_strategy_version
```

生产升级优先采用蓝绿重建：

```text
v1 继续提供查询
-> v2 后台全量重建
-> 升级期间新文档双写
-> 校验数据完整性、Recall@K、延迟和错误率
-> 灰度切换 v2
-> 保留 v1 回滚窗口
```

双写一边失败时，应根据当前在线查询版本判断用户可用性，并为另一边记录独立状态和补偿任务，不能用一个总状态掩盖部分失败。

### 离线 Recall 提升但线上满意度下降怎么排查？

`Recall@K` 只说明相关文档是否进入候选集，不保证关键证据排名、Prompt 上下文和最终生成质量。

线上需要按 `requestId` 记录：

```text
原始 Query 和改写 Query
召回 Chunk、排名和分数
Fusion 前后结果
Rerank 前后结果
最终进入 Prompt 的上下文
模型和 Prompt 版本
答案、引用、耗时和用户反馈
```

对差评请求逐层回放：

```text
是否召回关键证据
-> Fusion / Rerank 是否错误降权
-> 是否进入 Prompt
-> 是否被截断或重复内容挤占
-> LLM 是否正确使用证据
```

### 权限过滤应该放在哪里？

权限条件应进入检索本身，而不是全库召回后再过滤：

```text
认证用户
-> 计算可访问 tenant / knowledgeBase / namespace
-> 在授权集合内执行 TopK
-> Rerank
-> 构造 Prompt
```

全库 TopK 后过滤会造成错误拒答、敏感内容进入日志或缓存，以及响应时间、状态码等侧信道风险。

复杂 ACL 可优先转化为租户、知识库和权限组等稳定分区。多个授权分区应分别召回候选，合并去重后统一 Rerank，再取全局 TopK。

### 如何处理版本和有效时间？

不能简单只检索“当前最新版本”。规则检索需要区分：

```text
published_at
effective_at
expired_at
publish_status
business_scope
```

应根据问题对应的业务事件时间，检索当时正式发布且有效的版本。若当前规则包含过渡条款，可能需要先检索当前规则确定适用口径，再二次检索历史版本的具体条款。

### 如何控制重复证据和 Token 预算？

不能只按 Rerank 分数机械取前 K 条。相邻 Chunk 可能高度重叠，也可能分别包含一般规则和例外条件。

推荐流程：

```text
业务和权限过滤
-> Rerank
-> 相邻互补 Chunk 合并
-> 近重复内容去重
-> MMR 兼顾相关性和多样性
-> 在 Token 预算内选择最终证据
```

MMR 的核心是：

```text
候选分数 = Query 相关性 - 与已选证据的重复惩罚
```

但不能简单选择与 Top1 最不相似的内容，否则可能引入无关证据。规则类内容压缩时还必须保留主体、期限、条件、例外、证明要求和 `AND/OR/NOT` 关系，避免摘要改变业务语义。
