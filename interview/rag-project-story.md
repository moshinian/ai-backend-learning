# RAG 项目面试表达

## 1. 项目背景

云服务结算系统涉及产品、开发和测试等多个角色，历史需求、接口和设计文档比较分散。团队成员很难快速定位需要的信息，造成重复沟通和人工检索成本。

因此搭建内部 RAG 知识问答系统，让团队可以通过自然语言查询历史资料，并获得带来源依据的回答。

---

## 2. 两分钟项目介绍

这个项目来源于团队内部的实际问题。云服务结算系统涉及产品、开发和测试等多个角色，历史需求、接口和设计文档比较分散，人员很难快速定位信息，导致重复沟通和人工查找成本较高。因此，我搭建了一套面向内部资料的 RAG 知识问答系统。

系统主要分为知识库构建和在线问答两条链路。知识库构建阶段负责文档上传、解析、清洗、Chunk 切分、Embedding 生成以及向量入库。在线问答阶段接收用户问题，通过关键词和向量混合检索召回相关 Chunk，再将检索结果与问题组装成 Prompt，调用大模型生成回答，同时保留来源引用和会话记录。

技术架构上，Spring Boot 负责知识库、文档任务、用户会话和问答接口等业务能力；FastAPI 封装模型和向量化服务；PostgreSQL 负责业务数据和向量数据存储；Redis 用于缓存和部分任务状态管理。我负责系统整体架构和核心链路的搭建。

当前主要难点是检索准确率。部分问题在 `Top3` 中无法召回目标片段，扩大到 `Top5` 后召回率有所提升，但也增加了无关上下文。因此，扩大 `TopK` 只能作为现象验证，后续还需要从 Chunk 切分、查询改写、混合检索权重、元数据过滤、Rerank 和离线评测等方向继续优化。

---

## 3. 架构主线

```text
文档上传
  -> 解析与清洗
  -> Chunk 切分
  -> Embedding
  -> PostgreSQL / 向量存储
  -> 关键词与向量混合检索
  -> 融合排序 / Rerank
  -> Prompt 组装
  -> LLM 生成
  -> 引用与会话记录
```

## 4. 高频追问

### 为什么同时使用关键词和向量检索？

关键词检索适合订单编号、接口名称、错误码和专有名词等字面特征明确的内容，但对同义表达和自然语言改写不敏感。向量检索适合召回语义相近但表述不同的内容，但对编号、缩写和错误码等精确标识区分不够稳定。

混合检索利用关键词检索保证精确匹配，利用向量检索扩大语义召回，再通过融合排序或 Rerank 提升最终结果质量。

### 两种检索分数如何融合？

BM25 分数和向量相似度不在同一尺度，不能直接相加。可以先归一化后加权，也可以使用 RRF 按排名融合，再对候选结果执行 Rerank。

权重需要基于包含问题、相关 Chunk 和相关性等级的测试集，通过 `Recall@K`、MRR、NDCG 等指标调整，不能只根据个别问题凭感觉设置。

### 如何评价 RAG 效果？

1. 检索层：`Recall@K`、MRR、NDCG、证据完整性
2. 生成层：答案正确性、忠实度、引用准确性、完整性和拒答能力
3. 工程层：端到端延迟、首 Token 延迟、吞吐量、错误率、成本和可用性

需要把检索和生成分开评估，否则答案错误时无法判断问题发生在哪一层。

### 为什么不能整篇向量化？

整篇文档可能包含多个主题，向量会混合这些语义，使具体问题与文档整体向量的相似度下降。长文档还可能超过 Embedding 模型限制；召回后直接交给大模型也会增加 Token、延迟和上下文噪声。

### Chunk 如何切分？

优先按照标题、段落、列表、句子等自然边界切分，再通过最大 Token 数控制长度，并保留一定 Overlap，避免关键信息被边界截断。

参数不能固定。接口文档应尽量保证 URL、参数和返回值的关系不被破坏；规章制度需要保留章节层级；最终参数需要通过离线评测调整。

### Chunk 太大或太小有什么问题？

Chunk 太大会混合多个主题，降低检索定位精度，并增加 Token、延迟和上下文噪声。Chunk 太小会破坏事实和上下文完整性，例如把接口 URL、参数和返回值拆散，同时增加向量数量和检索碎片。

### 正确 Chunk 进入 TopK 后为什么仍回答错误？

可能原因包括：

1. 正确 Chunk 排名靠后
2. 片段证据不完整
3. 文档版本冲突
4. 上下文重复或组织混乱
5. Prompt 约束不足
6. 模型能力或生成参数不合适

诊断时先检查最终发送给模型的完整上下文是否足以支持答案，再判断是否属于模型问题。

### 什么是 Parent-Child Retrieval？

把文档同时切成较大的父 Chunk 和较小的子 Chunk。检索时使用语义集中的子 Chunk 提高定位精度，命中后通过 `parentId` 找回更完整的父级上下文，从而兼顾检索精度和证据完整性。

当前项目尚未实现该方案，应表达为针对 Chunk 信息被拆散问题的后续优化方向。
