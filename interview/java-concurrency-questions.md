# Java 并发面试问答

## 1. 线程池提交任务的顺序是什么？

### 简洁版

`ThreadPoolExecutor` 提交任务时，先创建核心线程；核心线程达到上限后，任务进入工作队列；队列满后再扩容到最大线程数；线程数达到最大值且队列也满时，触发拒绝策略。

### 深挖版

假设核心线程数为 4、最大线程数为 8、队列容量为 100，瞬间提交 110 个长任务：

```text
4 个由核心线程执行
100 个进入队列
4 个触发非核心线程扩容
2 个被拒绝
```

大队列会延迟扩容，并增加任务等待和内存风险；小队列会更早扩容与拒绝，也更容易把并发压力传递给下游。

## 2. 为什么生产环境不推荐直接使用 newFixedThreadPool？

### 简洁版

`newFixedThreadPool` 的线程数固定，但内部使用近似无界的 `LinkedBlockingQueue`。当提交速度长期超过消费速度时，任务会持续积压，导致等待时间失控和 OOM 风险。生产环境通常显式配置有界队列和拒绝策略。

### 深挖版

它不是任何场景都不能使用。固定数量的本地任务可以使用它；持续接收外部流量的长期运行服务需要明确容量边界、拒绝行为和监控指标。

## 3. CallerRunsPolicy 适合耗时后台任务吗？

### 简洁版

通常不适合。它会让提交任务的线程执行被拒绝的任务。如果提交者是 Tomcat 请求线程，三分钟的文档任务会让异步接口退化为同步接口，并可能耗尽 Web 工作线程。

### 深挖版

`CallerRunsPolicy` 可以对短任务形成提交方反压，但长任务会破坏线程池隔离。对于已持久化的后台任务，更适合使用 `AbortPolicy` 感知拒绝，记录 `RETRY_WAIT` 和下次重试时间，由调度器稍后处理。

## 4. 线程池参数为什么不能只按 CPU 核数配置？

### 简洁版

线程池并发还受数据库连接池、模型服务、网络和其他下游容量限制。线程数超过下游能力后，任务只会等待连接或限流许可，不会提升有效吞吐，反而增加内存、超时和上下文切换。

### 项目表达

RAG 文档处理包含 PDF 解析、Embedding 和向量入库。解析偏 CPU，Embedding 受模型服务并发限制，入库受数据库连接池和批量写入能力限制。因此需要按阶段资源模型配置并发，并通过压测观察吞吐、P95/P99 和错误率。

## 5. RAG 文档任务如何设计后台执行？

### 简洁版

上传接口只保存文件和任务记录并返回 `taskId`。后台 Worker 先取得本机执行许可，再通过短事务认领 `PENDING` 任务，写入 `process_token` 和租约；事务外执行解析、Embedding 和入库；执行期间心跳续租，最终带 Token 条件回写结果。

### 深挖版

多实例使用 `FOR UPDATE SKIP LOCKED` 避免抢占等待；线程池拒绝时进入带退避时间的 `RETRY_WAIT`；过期 Worker 无法通过 Token 校验回写；业务唯一键保证结果幂等；外部调用还需要稳定幂等键和调用记录。

## 6. 如何避免任务阶段间持续积压？

### 简洁版

状态表和批次 `LIMIT` 只能分批消费，不能真正反压。需要根据下游积压量和最老任务等待时间设置高低水位，动态降低或暂停上游解析速度；没有执行许可时任务保持 `PENDING`，而不是制造失败。

## 7. 如何处理反复失败的毒任务？

### 简洁版

区分可恢复和永久错误。可恢复错误进入 `RETRY_WAIT`，记录 `retry_count`、`last_error` 和 `next_retry_at`，使用指数退避加随机抖动；永久错误或超过最大次数后进入 `DEAD` 并告警，避免持续占用处理能力。
