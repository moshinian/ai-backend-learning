## 4. 什么是事务？为什么需要事务？

### 简洁版

事务是数据库用来保证一组操作作为整体执行的机制。它主要解决多个 SQL 操作中途失败后，数据只更新一部分，导致业务状态不完整的问题。

在结算系统里，比如账单批次入库、明细插入、批次状态更新，这类同库操作会放在一个本地事务里。但 ERP 推送、MQ 回写这类跨系统动作，不能只靠数据库事务保证一致性，需要结合状态机、幂等、重试补偿和对账实现最终一致。

### 深挖版

事务解决的是“一个业务动作由多条数据库变更组成时，如何避免中间状态暴露或部分成功”的问题。比如账单解析入库时，批次表、明细表、状态字段必须一起形成合法状态。如果明细插入失败但批次状态已经更新为成功，后续结算任务就会基于错误数据继续处理。

但事务不是万能的。跨系统调用，比如 ERP 推送或 MQ 发送，已经不在同一个数据库事务资源里，不能依赖一个本地事务回滚全部链路。工程上通常会把本地状态落库作为事实来源，通过状态机记录阶段，通过唯一键和业务流水号保证幂等，通过补偿任务和对账处理失败状态。

## 5. 脏读、不可重复读、幻读分别是什么？

### 简洁版

脏读是一个事务读到了另一个事务还没提交的数据，如果对方回滚，这个数据其实从未真正生效。不可重复读是同一个事务里两次读取同一行数据，因为其他事务提交了修改，导致内容变了。幻读是同一个事务里按相同条件做范围查询，两次查询的结果集发生变化，比如多出或少了符合条件的记录。

区别是：脏读读的是未提交数据；不可重复读关注同一行内容变化；幻读关注范围结果集变化。

### 项目表达

在结算系统里，不会只依赖隔离级别解决所有并发问题。批次处理会用事务保证本地原子性，用状态机约束状态流转，用条件更新、乐观锁、唯一约束和幂等键避免重复处理。批处理场景还会通过批次号或数据打标固定本次处理范围，避免新数据影响当前批次。

## 6. MySQL 默认隔离级别是什么？为什么不是 Serializable？

### 简洁版

MySQL InnoDB 默认隔离级别是可重复读。数据库不默认使用 Serializable，是因为串行化虽然隔离性最强，但会明显降低并发能力。实际业务里很多事务操作的是不同用户、不同订单、不同批次，它们之间没有数据冲突，不应该被强行排队。

InnoDB 默认选择可重复读，是在一致性和性能之间折中。普通查询通过 MVCC 读取一致性快照，减少读写阻塞；真正有并发冲突风险的场景，再通过条件更新、乐观锁、唯一约束或 `select ... for update` 做精确控制。

## 7. MVCC 解决什么问题？大概怎么工作？

### 简洁版

MVCC 是多版本并发控制。InnoDB 不会让普通读都加锁，而是通过隐藏事务 ID、undo log 和 Read View 判断当前事务应该看到哪个版本。事务生成一致性视图后，普通查询如果发现最新版本是其他事务之后提交的、对自己不可见，就会沿 undo log 找到旧版本返回。这样可以实现可重复读，同时减少读写互相阻塞。

### 深挖版

MVCC 不是复制整张表，也不是保存查询结果集，而是为行数据保留历史版本，并用 Read View 做可见性判断。普通 `select` 属于快照读，主要通过 MVCC 保持一致视图；`select ... for update`、`update`、`delete` 属于当前读，需要读最新版本并加锁。

## 8. undo log、redo log、binlog 分别解决什么问题？

### 简洁版

`undo log` 主要用于事务回滚和 MVCC 历史版本读取。事务修改数据前会记录旧值，失败时可以根据 undo log 回滚，快照读也可以沿 undo log 找旧版本。

`redo log` 是 InnoDB 引擎层日志，主要用于崩溃恢复，保证事务提交后即使数据页还没刷盘，也能恢复已提交的数据。

`binlog` 是 MySQL Server 层日志，主要用于主从复制、时间点恢复和变更订阅。redo log 保证本机崩溃恢复，binlog 保证变更能被复制和恢复，两者通过两阶段提交协调一致。

## 9. MySQL 事务主线综合回答

### 1 到 2 分钟版

MySQL 事务主要解决的是：一组数据库操作需要作为一个整体执行，避免部分成功导致数据处于不合法状态。比如结算系统里，批次表、明细表、状态更新这类同库操作，要么一起成功，要么一起回滚。

为了支持事务，InnoDB 会用 `undo log` 和 `redo log`。`undo log` 记录修改前的数据，事务失败时可以回滚，同时也为 MVCC 提供历史版本；`redo log` 用于崩溃恢复，保证事务提交后即使数据页还没刷盘，也能恢复已提交的数据。

事务并发时会有脏读、不可重复读、幻读这些问题，所以数据库提供隔离级别。MySQL InnoDB 默认是可重复读，它不是靠所有读都加锁，而是通过 MVCC 实现一致性快照读。事务生成 `Read View` 后，普通查询会根据可见性规则判断当前行版本能不能读；如果最新版本不可见，就沿着 `undo log` 找到旧版本，因此可以减少读写阻塞。

`binlog` 是 MySQL Server 层日志，主要用于主从复制、数据恢复和变更订阅。它和 `redo log` 职责不同：`redo log` 保证本机崩溃恢复，`binlog` 保证变更能被复制和恢复。为了避免主库事务提交了但 binlog 没记录，MySQL 需要通过两阶段提交协调两者。
