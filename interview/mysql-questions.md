## 4. 什么是事务？为什么需要事务？

### 简洁版

事务是数据库用来保证一组操作作为整体执行的机制。它主要解决多个 SQL 操作中途失败后，数据只更新一部分，导致业务状态不完整的问题。

在结算系统里，比如账单批次入库、明细插入、批次状态更新，这类同库操作会放在一个本地事务里。但 ERP 推送、MQ 回写这类跨系统动作，不能只靠数据库事务保证一致性，需要结合状态机、幂等、重试补偿和对账实现最终一致。

### 深挖版

事务解决的是“一个业务动作由多条数据库变更组成时，如何避免中间状态暴露或部分成功”的问题。比如账单解析入库时，批次表、明细表、状态字段必须一起形成合法状态。如果明细插入失败但批次状态已经更新为成功，后续结算任务就会基于错误数据继续处理。

但事务不是万能的。跨系统调用，比如 ERP 推送或 MQ 发送，已经不在同一个数据库事务资源里，不能依赖一个本地事务回滚全部链路。工程上通常会把本地状态落库作为事实来源，通过状态机记录阶段，通过唯一键和业务流水号保证幂等，通过补偿任务和对账处理失败状态。

## 5. 脏读、不可重复读、幻读分别是什么？

### 简洁版

脏读是一个事务读到了另一个事务还没提交的数据，如果对方回滚，这个数据其实从未真正生效。不可重复读是同一个事务里两次读取同一行数据，因为其他事务提交了修改，导致内容变了。幻读是同一个事务里按相同条件做范围查询，两次查询的结果集发生变化，比如多出或少了符合条件的记录。

区别是：脏读读的是未提交数据；不可重复读关注同一行内容变化；幻读关注范围结果集变化。

### 项目表达

在结算系统里，不会只依赖隔离级别解决所有并发问题。批次处理会用事务保证本地原子性，用状态机约束状态流转，用条件更新、乐观锁、唯一约束和幂等键避免重复处理。批处理场景还会通过批次号或数据打标固定本次处理范围，避免新数据影响当前批次。

## 6. MySQL 默认隔离级别是什么？为什么不是 Serializable？

### 简洁版

MySQL InnoDB 默认隔离级别是可重复读。数据库不默认使用 Serializable，是因为串行化虽然隔离性最强，但会明显降低并发能力。实际业务里很多事务操作的是不同用户、不同订单、不同批次，它们之间没有数据冲突，不应该被强行排队。

InnoDB 默认选择可重复读，是在一致性和性能之间折中。普通查询通过 MVCC 读取一致性快照，减少读写阻塞；真正有并发冲突风险的场景，再通过条件更新、乐观锁、唯一约束或 `select ... for update` 做精确控制。

## 7. MVCC 解决什么问题？大概怎么工作？

### 简洁版

MVCC 是多版本并发控制。InnoDB 不会让普通读都加锁，而是通过隐藏事务 ID、undo log 和 Read View 判断当前事务应该看到哪个版本。事务生成一致性视图后，普通查询如果发现最新版本是其他事务之后提交的、对自己不可见，就会沿 undo log 找到旧版本返回。这样可以实现可重复读，同时减少读写互相阻塞。

### 深挖版

MVCC 不是复制整张表，也不是保存查询结果集，而是为行数据保留历史版本，并用 Read View 做可见性判断。普通 `select` 属于快照读，主要通过 MVCC 保持一致视图；`select ... for update`、`update`、`delete` 属于当前读，需要读最新版本并加锁。

## 8. undo log、redo log、binlog 分别解决什么问题？

### 简洁版

`undo log` 主要用于事务回滚和 MVCC 历史版本读取。事务修改数据前会记录旧值，失败时可以根据 undo log 回滚，快照读也可以沿 undo log 找旧版本。

`redo log` 是 InnoDB 引擎层日志，主要用于崩溃恢复，保证事务提交后即使数据页还没刷盘，也能恢复已提交的数据。

`binlog` 是 MySQL Server 层日志，主要用于主从复制、时间点恢复和变更订阅。redo log 保证本机崩溃恢复，binlog 保证变更能被复制和恢复，两者通过两阶段提交协调一致。

## 9. MySQL 事务主线综合回答

### 1 到 2 分钟版

MySQL 事务主要解决的是：一组数据库操作需要作为一个整体执行，避免部分成功导致数据处于不合法状态。比如结算系统里，批次表、明细表、状态更新这类同库操作，要么一起成功，要么一起回滚。

为了支持事务，InnoDB 会用 `undo log` 和 `redo log`。`undo log` 记录修改前的数据，事务失败时可以回滚，同时也为 MVCC 提供历史版本；`redo log` 用于崩溃恢复，保证事务提交后即使数据页还没刷盘，也能恢复已提交的数据。

事务并发时会有脏读、不可重复读、幻读这些问题，所以数据库提供隔离级别。MySQL InnoDB 默认是可重复读，它不是靠所有读都加锁，而是通过 MVCC 实现一致性快照读。事务生成 `Read View` 后，普通查询会根据可见性规则判断当前行版本能不能读；如果最新版本不可见，就沿着 `undo log` 找到旧版本，因此可以减少读写阻塞。

`binlog` 是 MySQL Server 层日志，主要用于主从复制、数据恢复和变更订阅。它和 `redo log` 职责不同：`redo log` 保证本机崩溃恢复，`binlog` 保证变更能被复制和恢复。为了避免主库事务提交了但 binlog 没记录，MySQL 需要通过两阶段提交协调两者。

## 10. 快照读、当前读、行锁、间隙锁、Next-Key Lock 分别是什么？

### 简洁版

快照读主要是普通 `select`，InnoDB 在 RR 下通过 MVCC、Read View 和 undo log 读取一致性历史视图，一般不加锁。当前读读取最新数据，并且通常会加锁，比如 `select ... for update`、`update`、`delete`。

行锁解决多个事务同时修改同一条记录的问题；间隙锁锁住索引记录之间的范围，防止其他事务插入新记录；Next-Key Lock 是行锁加间隙锁，常用于 RR 下范围当前读防止幻读。

### 深挖版

普通快照读和当前读要分开看。RR 下普通 `select` 的幻读问题主要由 MVCC 的一致性视图处理，不是靠 Next-Key Lock。当前读必须基于最新数据做判断，如果是范围查询，就可能需要锁住已有记录和索引间隙，避免其他事务插入会影响判断的新记录。

需要注意，InnoDB 的锁依赖索引访问路径。唯一索引等值命中时可能退化为记录锁；范围查询、非唯一索引查询更容易涉及间隙锁或 Next-Key Lock；没有合适索引时，扫描范围变大，锁住的记录也可能变多。

## 11. 为什么不能先 select 一批 INIT 数据再直接处理？

### 简洁版

先 `select` 一批 `INIT` 数据再处理有并发问题，因为查询候选数据和声明处理权不是一个原子动作。多个节点可能同时查到同一批 `INIT` 数据，导致重复处理。

更稳的做法是用状态条件更新来抢占，比如：

```sql
update settlement_order
set status = 'PROCESSING'
where id = ?
  and status = 'INIT';
```

通过影响行数判断是否抢占成功。批量场景可以先查候选 ID，再用 `update ... where id in (...) and status='INIT'` 批量抢占，最后只处理真正抢占成功的数据。

### 深挖版

分片字段可以减少不同 worker 扫描同一批数据，例如按 `shard_no` 拆分任务范围，但分片只是降低冲突、提升吞吐，不能替代原子抢占。只要存在多节点、补偿、重跑或同分片并发，就仍然要靠状态条件更新确认处理权。

也不建议把 `select ... for update` 和长耗时业务处理放在一个大事务里。结算、文件解析、ERP 推送、MQ 回写这类操作耗时长，如果一直持有数据库锁和连接，会明显降低吞吐并放大阻塞。更合理的是短事务抢占任务，事务提交后执行耗时处理，最后再用短事务回写结果。

## 12. PROCESSING 状态卡住怎么办？

### 简洁版

`PROCESSING` 不能设计成永久状态，而应该设计成带租约的处理中状态。worker 抢占任务时记录 `worker_id`、`process_start_time`、`heartbeat_time`、`retry_count`。后台补偿任务定期扫描长时间未心跳的数据，用条件更新恢复为 `INIT` 或 `RETRY`。

但超时不代表一定失败，所以重跑必须配合幂等控制，比如状态机条件更新、唯一业务键、ERP 幂等流水号和 MQ 消费去重，避免重复结算。

### 深挖版

补偿任务不应该通过停业务、禁止 SQL 的方式修复卡住数据，而应该在线扫描超时未续约的任务。恢复时必须带上状态、时间或版本条件，避免误改刚刚完成或刚刚续约的数据。

如果任务链路比较长，建议引入 `process_token`、`attempt_id` 或 `batch_id`。抢占时生成本次处理令牌，后续心跳、成功、失败回写都必须带上这个令牌：

```sql
update settlement_order
set status = 'SUCCESS'
where id = ?
  and status = 'PROCESSING'
  and process_token = ?;
```

这样即使旧 worker 处理很慢，任务被补偿回收并被新 worker 抢走，旧 worker 最后的回写也会因为 token 不匹配而影响 0 行，不会覆盖新状态。

## 13. ERP 推送成功但本地还没改 SUCCESS 就宕机，怎么处理？

### 简洁版

这是本地事务和外部系统调用之间的不确定状态。本系统调用 ERP 成功后，如果还没更新本地 `SUCCESS` 就宕机，本地无法只靠数据库判断 ERP 是否已经处理。

我会做三层保障：第一，推送 ERP 时带业务唯一键或 `request_id`，让 ERP 支持幂等，同一个 key 内容一致时重复请求返回已成功，内容不一致则拒绝并告警；第二，本地维护 ERP 推送记录表，记录请求号、payload hash、状态、重试次数和返回流水；第三，如果 ERP 支持查询，就先按请求号查询结果，确认已成功后补偿本地状态，否则再幂等重推。财务场景还需要对账兜底。

### 深挖版

本地事务不能覆盖 ERP 这种外部系统，所以要把外部调用设计成可重试、可查询、可对账。推送前先落本地调用记录，发起调用时标记为 `SENDING`，收到明确成功后标记为 `SUCCESS`。如果超时或宕机导致结果不确定，可以进入 `UNKNOWN` 或保持待确认状态，后续补偿任务优先查询 ERP 结果。

ERP 侧的幂等语义不能只是简单拒绝重复请求。更合理的是：同一幂等键且内容一致，返回已成功；同一幂等键但内容不一致，拒绝并告警。这样本系统重试时不会把“重复成功”误判为失败。
