# Redis 面试问答

## 1. 为什么 JVM 内的 synchronized 不能解决多实例任务并发？

### 简洁版

`synchronized` 只约束同一个 JVM 内竞争同一把锁的线程。多实例部署时，每个实例都有自己的锁对象，实例 A 加锁不会阻止实例 B、C 同时处理同一任务。因此多实例任务抢占需要数据库条件更新、Redis 分布式锁或 MQ 分区等跨进程机制。

## 2. Redis 分布式锁如何正确加锁？

### 简洁版

通常使用：

```redis
SET lock_key unique_value NX PX 30000
```

`NX` 表示 key 不存在才写入，`PX` 表示设置毫秒级过期时间，`unique_value` 用于标识当前持锁者。加锁和设置过期时间必须在一条命令中原子完成，不能拆成 `SETNX` 和 `EXPIRE`。

## 3. 为什么释放锁不能直接 DEL？

### 简洁版

因为当前实例的锁可能已经过期，并被其他实例重新获取。旧实例如果直接 `DEL`，会误删新实例的锁。释放锁必须用 Lua 脚本原子判断 value 是否等于自己的 `unique_value`，匹配时才删除。

## 4. 看门狗续期解决什么问题？

### 简洁版

看门狗是后台续期任务。长任务执行时间超过锁租期时，它定期校验锁 value 仍属于自己，然后续期，避免锁提前过期。但看门狗只能说明续期逻辑还在运行，不能证明业务任务健康，因此还需要任务最大执行时间、业务心跳、超时回收和 `process_token`。

## 5. 为什么续期也要校验 unique_value？

### 简洁版

如果旧锁已过期，新实例已经获得同名锁，旧实例的看门狗如果不校验 value，就可能给新实例的锁续期，干扰新持锁者。续期必须和释放一样，先确认 value 属于自己。

## 6. Redis 锁和 process_token 的区别是什么？

### 简洁版

Redis 锁是临时互斥，解决多个实例短时间内不要同时进入临界区；`process_token` 是数据库中的处理权标识，解决当前 Worker 是否还有资格推进任务状态。Redis 锁丢失或过期时，最终数据库回写仍要通过 `process_token` 条件更新兜底。

## 7. 为什么不能只靠 Redis 锁保证任务只处理一次？

### 简洁版

Redis 锁可能过期、续期失败、被误删或在 Redis 故障切换中失效。即使锁正常，也无法回滚已经发生的外部副作用。任务只处理一次需要数据库状态机、条件更新、`process_token`、业务幂等键、下游幂等和补偿对账共同保证。

## 8. ERP 调用为什么不能用 process_token 做幂等键？

### 简洁版

`process_token` 每次认领都会变化。如果用它作为 ERP 幂等键，同一结算单重试时会变成不同请求，可能重复记账。ERP 幂等键应该使用稳定业务身份，例如 `settlement_order_id` 或 `posting_request_id`，并绑定请求内容 hash。

## 9. 外部调用超时后如何处理？

### 简洁版

网络超时只代表调用方不知道结果，不代表外部系统失败。应先使用原幂等键查询外部请求状态；查不到明确结果时，用同一个幂等键重试；仍不确定则进入 `CONFIRMING` 或 `RETRY_WAIT`，通过延迟补偿和对账兜底。不能直接标记失败，也不能换新幂等键重试。

## 10. 任务处理权应以 Redis 锁还是数据库条件更新为主？

### 简洁版

应以数据库条件更新和状态机为主。任务状态事实在数据库里，`UPDATE ... WHERE status='INIT'` 可以原子抢占，后续回写带 `process_token` 防止旧 Worker 覆盖。Redis 锁适合作为辅助互斥，降低并发进入临界区的概率，但不能替代数据库状态机。
