### 问题 3：Chunk 切片策略

**问题**

你们按固定字符数还是段落切？为什么不用 Word 自带标题层级？如何保证语义连贯？

**现场回答或目标回答**

现场说明采用固定窗口 + overlap，末端尽量落在自然标点；澄清系统对标题行和正文有区分存储；窗口大小根据文档情况动态调整，并通过召回 TopK 效果判断。

**评价**

固定窗口、overlap、自然边界和标题 / 正文区分能解释当前实现，但“通过 TopK 再调窗口”会显得前置标准不足。面试官期待的是结构优先、长度兜底、语义增强和评测闭环。

**改进版**

> 我会把 chunk 分成三层：第一层结构优先，利用标题层级、段落、列表、表格边界保留天然语义单元；第二层长度兜底，过长内容按 token 窗口切并保留 overlap；第三层语义增强，对结构不明显的长段落用相邻句子或段落 embedding 相似度下降来找候选切点。最后通过 Recall@K、MRR、引用准确性和人工样例验证参数。

**后续任务**

继续补语义切分、表格 chunk 和 chunk 评测口径。

### 问题 4：语义切分

**问题**

固定窗口可能导致语义断裂，有没有办法做语义切割？

**现场回答或目标回答**

现场提出可以基于 embedding 做语义分块，或用大模型做预切分；针对接口文档等特殊场景定制切块规则。

**评价**

能想到方向，但缺工程级实现。语义切分不能只说“大模型帮切”，需要讲清基础单元、相似度计算、候选切点、token 约束、结构保护和回退机制。

**改进版**

> 语义切分可以先把段落拆成句子或小段落作为基础单元，对相邻单元或相邻单元组生成 embedding，计算 cosine similarity。相似度明显下降的位置作为候选切点，再结合 min_tokens、max_tokens、overlap 和标题 / 表格边界做约束。大模型预切分适合离线索引阶段，但要控制成本，并保留规则切分兜底。

**后续任务**

在 `interview/ai-application-questions.md` 或正式 RAG 笔记中补“语义切分工程实现”。

### 问题 5：混合检索与结果融合

**问题**

混合检索怎么做？关键词匹配是什么？双路召回结果如何融合？

**现场回答或目标回答**

现场说明除语义检索外，还使用 SQL 内置函数做精准词匹配，用于股票代码、编号等弱语义信息；双路召回后采用归一化权重，或进一步用 Rerank。

**评价**

关键词兜底和弱语义场景讲得住，但需要更清楚地区分召回层、融合层和重排层。SQL 精确匹配、全文检索、BM25、RRF、Rerank 不能混在一层讲。

**改进版**

> Hybrid Retrieval 发生在召回层，一路用向量检索解决语义相似，一路用关键词 / 全文检索解决编号、代码、专有名词的精确匹配。融合可以用 RRF 或归一化加权，把不同召回路径的候选合并。Rerank 是召回后的精排，对合并后的 TopK 候选重新打相关性分数。

**后续任务**

补 BM25 / keyword recall / RRF / rerank 的边界图。

### 问题 6：Rerank 内部机制

**问题**

RAG 模型中的重排算法内部机制是什么？

**现场回答或目标回答**

现场把 Rerank 泛化为语料训练、编码器-解码器和向量转换，没有精准命中 Rerank 的 query-document pair 交互机制。

**评价**

这是本场核心失分点。面试官想确认你是否知道 bi-encoder 召回和 cross-encoder 重排的差异，以及 reranker 为什么能比向量召回做更细粒度判断。

**改进版**

> 第一阶段召回常用 bi-encoder，把 query 和 chunk 分别编码成向量，适合快速召回。Rerank 常用 cross-encoder，把 query 和候选 chunk 拼成一对输入模型，让模型直接判断这一对是否相关，输出相关性分数。它慢一些，但能利用 query 和文本之间的 token 级交互，所以适合对 TopK 候选做精排。

**后续任务**

把 bi-encoder、cross-encoder、训练数据、hard negative 和私有语料微调补成可口述答案。

### 问题 7：私有语料排序失效

**问题**

如果数据私有性强，开源模型效果差，双路召回后最相关段落仍排在末尾，怎么办？

该问题需要分层排查：正确片段是否进入候选集、融合策略是否压低了分数、reranker 是否不适配私有语料、是否缺业务规则特征和评测数据。

**改进版**

> 我会先确认正确片段是否进入候选集。如果已经进入但排在后面，问题在 fusion 或 rerank。短期可以加入业务规则特征，例如标题命中、关键词精确匹配、文档类型、版本有效期、章节层级和实体匹配，对分数做校正。中期建立 query、正例 chunk、hard negative 的评测集，微调 reranker。长期把用户点击、人工采纳和反馈结果沉淀为训练数据，形成检索优化闭环。

## 5. 跨题错误模式

1. 真实强项是工程化，但简历中出现 RAG、算法项目、Embedding、Rerank 等关键词后，面试官会自然追问算法底层。
2. 文档解析不能只回答“支持 Word / PDF”，要回答文档结构如何进入 chunk 和检索。
3. Chunk 不能只回答固定窗口和 overlap，要先讲结构边界，再讲长度兜底和语义增强。
4. Rerank 不能泛化为“再排序模型”，要讲清 cross-encoder 对 query-document pair 打分。
5. 私有语料排序失效时，不能只说换模型或重新训练，要先做候选集诊断，再做业务规则、评测集、hard negative 和微调。
6. 金融 / ERP / 结算类数据表达中，不能轻易说“丢弃”，要先判断数据是否允许丢弃。
7. 前沿算法关注不能只说“看推文”，要有论文、源码、benchmark、小实验的固定学习机制。
