## 3. 逐题复盘

### 3.1 RAG 知识问答系统架构

**面试官原问题**

> 请介绍 RAG 知识问答系统的架构。

**我的现场回答**

> RAG 知识系统分为知识库创建和在线问答。知识库创建包含集成、解析、Chunk 切分、Embedding 和向量化存储；在线问答包含用户提问、混合检索、Rerank 排序、大模型生成、会话引用和保存。Spring Boot 负责业务接口和任务管理，FastAPI 封装模型服务，PostgreSQL 作为数据库，Redis 作为缓存。

**评价：6/10**

回答正确的部分：

1. 能拆分知识库构建和在线问答两条主链路。
2. 能说明 Spring Boot、FastAPI、PostgreSQL 和 Redis 的大致职责。

主要问题：

1. 偏功能罗列，缺少业务目标、数据流和工程保障。
2. 当时把归一化加权融合称为 Rerank。
3. 当前尚未实现独立 Rerank，不能包装成现有能力。
4. “技术栈”多次口误为“技术债”。

**更好的回答**

> 这个系统用于解决企业文档分散、人工检索效率低，以及通用大模型缺少企业私域知识的问题。系统分为知识库构建和在线问答两条链路。知识库构建阶段负责文档接入、解析、清洗、Chunk 切分、Embedding 和向量入库；在线问答阶段通过 PostgreSQL 全文检索和 pgvector 语义检索召回候选，再进行归一化加权融合，选取相关 Chunk 组装 Prompt，调用大模型生成答案，并保存来源引用和会话记录。Spring Boot 负责业务接口、任务编排和状态管理，FastAPI 封装模型服务，PostgreSQL 保存业务与向量数据，Redis 用于缓存。知识库构建任务通过状态落库、幂等和重试支持失败恢复。

### 3.2 Chunk 如何切分

**面试官原问题**

> Chunk 是如何切片的？

**我的现场回答**

> 通用兜底方法是先设置固定 Chunk 窗口，200 或 500 均可，Overlap 取 50。到达窗口末端时判断是不是以逗号、表格符号、句号、空格、换行或段落符结尾；如果不是，就在窗口附近寻找最近的段落符作为最终端点。具体方法要依据文档格式和用户问答粒度决定，例如接口文档倾向于将 URL 和接口参数放入一个 Chunk，小文档可以依据标题切分段落，最终以离线评测效果为主。

**评价：7.5/10**

回答正确的部分：

1. 覆盖固定窗口、Overlap、自然边界和文档类型适配。
2. 知道参数不能脱离实际问答效果，需要离线评测。

主要问题：

1. 没有说明 200、500 和 50 的单位是字符还是 Token。
2. “均可”显得参数缺少实验依据。
3. 无限制向后找边界可能突破最大输入长度。

**更好的回答**

> 通用策略先设置最大 Token 窗口和 Overlap，到达边界时优先在窗口内向前寻找段落、换行或句子结束位置，尽量保持语义完整，同时保留硬性最大长度。结构化文档采用领域策略，例如接口文档尽量把 URL、参数、返回值和说明放在同一 Chunk。初始参数只是实验起点，最终要通过标注问答集比较 Recall@K、回答正确性和引用准确性。

### 3.3 混合检索的实现

**面试官原问题**

> 混合检索是怎么做的？

**我的现场回答**

> 混合检索包括语义相似性检索和精准词检索，Rerank 归一化语义检索和精准词检索的相似度权重。

后续确认：

1. 精准词检索实际使用 PostgreSQL 全文检索能力，见过 `to_tsvector`、`tsquery` 和 `@@`。
2. 语义检索使用向量相似度。

**评价：5/10**

主要问题：

1. 归一化加权属于 Fusion，不是 Rerank。
2. PostgreSQL 全文检索的词向量不是 Embedding 向量。

**更好的回答**

> 在线问答并行执行两路召回：一路通过 pgvector 做语义检索，另一路通过 PostgreSQL 全文检索匹配关键词和专有名词。两路分数先归一化再加权融合，或者使用 RRF 按排名融合。目前尚未接入独立 Rerank 模型。

### 3.4 混合检索如何优化

**面试官原问题**

> 混合检索可以怎么优化？

**我的现场回答**

> 一方面优化生成召回，例如使用更好的 Prompt、调整 Chunk 切分和相似权重；另一方面优化大模型生成能力，例如调整模型温度或者更换大模型。

**评价：4/10**

主要问题：

1. 将检索层和生成层混在一起。
2. Prompt、模型温度和 LLM 选型不属于混合检索优化。
3. “生成召回”不是准确术语。

**更好的回答**

> 我会从四层优化。第一层分别优化向量检索和全文检索，包括 Chunk、Embedding、查询改写、元数据过滤和各自的 TopK；第二层通过归一化加权或 RRF 优化结果融合；第三层对融合候选接入 Cross-Encoder Rerank 做二阶段重排；第四层建立标注问答集，用 Recall@K、MRR、NDCG 和延迟指标验证改动。

### 3.5 全文检索与向量分数如何融合

**面试官追问**

> 两路检索分数为什么不能直接相加？权重如何确定？

**我的回答**

> 因为两者计算概率的方式不一样。可以使用归一化权重，通过离线评价结果寻找最合适的权重参数。

**评价：6/10**

主要问题：

1. 两路输出不是概率，而是含义、尺度和分布不同的相关性分数。
2. 融合应针对每个候选文档计算，不能先分别求和。

**修正公式**

```text
score(d) =
alpha * norm(vectorScore(d))
+ (1 - alpha) * norm(keywordScore(d))
```

`alpha` 通过验证集上的检索指标选择。若分数归一化不稳定，可以使用 RRF 按排名融合。

### 3.6 Recall@K、MRR 和 NDCG

**面试官追问**

> 如何评价混合检索效果？

**我的现场回答**

> Recall@K 是正确文档在召回排名中的平均排位，最高值是 1。MRR 会设置完全一致、基本一致和不一致的分值，然后加权累加。

**评价：2/10**

两个定义均有混淆：

1. `Recall@K` 衡量相关文档是否或有多少进入 TopK，不关心其具体名次。
2. MRR 是第一个相关结果排名倒数的平均值。
3. 分级相关性和排名折损更接近 NDCG。

复盘验证：

```text
正确文档排第 4：
Recall@5 = 1
RR = 1/4
```

### 3.7 Cross-Encoder Rerank

**复盘中的关键问题**

> Cross-Encoder 是否使用大模型能力？工程上如何落地？

**复盘后的理解**

1. Embedding 模型负责大规模快速召回。
2. Cross-Encoder 将 `query + chunk` 联合编码，输出相关性分数。
3. Reranker 对少量候选进行精细排序，不负责生成答案。
4. ChatGPT、DeepSeek 等生成模型根据最终 Chunk 生成答案。
5. Rerank 无法找回第一阶段没有召回的 Chunk。

工程链路：

```text
混合召回 Top20~50
-> Fusion 和去重
-> FastAPI 批量执行 Cross-Encoder
-> 选择 Top3~5
-> 组装 Prompt
-> LLM 生成
```

**复盘后评价：8/10**

当前已能区分召回问题、排序问题和生成问题。

### 3.8 Prompt 如何优化和组装

**面试官原问题**

> Prompt 如何优化组装？调整 Prompt 后，原先排序靠前的内容会不会变成靠后？

**我的现场回答**

> 调整 Prompt 的确会导致最终效果波动。从全局角度可以根据整体离线评测结果调整 Prompt；从局部角度可以针对不同场景的问答使用不同 Prompt。

**评价：6/10**

回答正确的部分：

1. 知道 Prompt 变更需要离线评测。
2. 知道不同业务意图可以使用不同模板。

主要问题：

1. 如果 Prompt 只用于最终生成，它不会改变检索排序。
2. 只有 Query Rewrite 或 LLM Rerank 使用 Prompt 时，才可能影响召回或排序。
3. 缺少模板结构、版本管理和回归机制。

推荐结构：

```text
角色和目标
-> 回答规则
-> 检索上下文
-> 必要会话历史
-> 用户问题
-> 输出和引用格式
```

不同场景应通过意图分类路由到少量、可版本化、可评测的 Prompt 模板，不能为单个问题持续增加特例。

### 3.9 用户问答如何嵌入 Agent

**面试官原问题**

> 如何将用户问答嵌入 Agent？如何组装到 AI Pipeline？有没有参与过 Agent 类开发？

**我的现场回答**

> 实际工程中可能没有做过这类事情，也可能是我的理解有误。项目早期使用离线 Embedding 和 LLM 模型，本地模型使用过 Qwen 系列，后来发现效果不好，改为通过 API 调用公司私有部署的 DeepSeek 模型。

**评价：3/10**

回答正确的部分：

1. 没有虚构 Agent 生产经验。

主要问题：

1. 模型选型没有回答 Agent 编排。
2. 当时没有区分固定 Pipeline 与动态 Agent。

**更好的回答**

> 我目前没有完整的 Agent 生产项目，实际完成的是固定 RAG Pipeline。用户问题进入系统后，依次经过检索、Prompt 组装、模型生成、引用绑定和会话保存，步骤由程序预定义。如果升级为 Agent，需要将问题、会话历史、工具结果和当前步骤放入统一 State，由模型根据状态选择知识库、数据库或外部 API 等工具。系统执行工具后把 Observation 写回 State，再由模型判断继续调用、请求补充信息还是生成最终答案。工程上还要限制最大步骤数、权限、超时、重试和高风险操作。

复盘后的判断：

> Agent 的关键不是工具数量，而是是否形成受约束的“决策 -> 执行 -> 观察 -> 再决策”闭环。

### 3.10 结算系统业务场景和架构

**面试官原问题**

> 介绍 DERP 结算系统架构、细分场景、技术栈和订单规模。

**我的现场回答**

> 系统负责荣耀云服务结算，手机 App 相关结算由系统处理。业务分为四类：2C、普通 2B、商推和流水。2C 包括游戏、主题等应用，会产生大量订单，系统负责订单集成、客户主数据与合同匹配、应收应付、开票、辅账和核销；普通 2B 例如手机预装，订单较少，通常提供汇总表；商推虽然属于 2B，但订单量也很大，因此单独处理；流水用于和订单核销。
>
> Spring Boot 构成系统微服务，包括订单集成、规则匹配、批处理、状态管理、MQ 和幂等。技术组件包括 MySQL、MQ、Redis、文件存储、网关、PaaS 中间件、平台定制的 K8s 和日志管理。订单规模约每天 600 万笔。

**评价：6.5/10**

回答正确的部分：

1. 四类业务场景真实清楚。
2. 给出了日订单量约 600 万的量化数据。
3. 技术背景和基础设施信息丰富。

主要问题：

1. 业务、架构、技术栈和基础设施混在一起。
2. 缺少清晰的上下游、数据流和同步异步边界。
3. 多次将“技术栈”口误为“技术债”。

建议结构：

```text
系统定位
-> 四类业务场景
-> 上游接入
-> 规则与结算处理
-> 财务输出和流水核销
-> 异步化、状态机和失败恢复
-> 规模与瓶颈
```

### 3.11 性能瓶颈和数据库演进

**面试官原问题**

> 系统目前的性能瓶颈在哪里？数据库采用了什么分库分表策略？

**我的现场回答**

> 业务希望一个批处理过程在 30 分钟内完成，但系统实际需要约 1 小时。通过日志发现 Java 代码执行耗时不高，大部分时间花在数据库查询上。系统早期订单不多；日订单量达到约 100 万后，先按业务日期对业务表做分区，根据结算周期设置一周或一个月的分区，并通过历史库实现冷热分离。后来数据库逐渐难以支撑业务量，又按年份拆库，也做过索引优化。部分问题受历史业务模型限制，需要重构才能进一步优化。

后续补充：

1. 单条 SQL 本身不慢，主要是累计数据库访问耗时较高。
2. 客户信息采用批量查询。
3. 订单状态通过一条 SQL 批量更新。
4. 每批调整为 500~1000 条后，整体耗时仍未明显下降。

**评价：6/10**

回答正确的部分：

1. 有真实的数据库演进过程。
2. 能说明业务时间目标与实际完成时间的差距。
3. 没有把索引包装成万能方案。

主要问题：

1. JDBC 调用阶段耗时高，不等于数据库计算能力不足。
2. 日订单量 600 万的平均 TPS 不高，压力还可能来自集中批处理、大量关联、总扫描量、锁竞争或历史数据膨胀。
3. 按年份拆库更接近数据生命周期拆分，未必解决当年流量的水平扩展问题。
4. 当前尚未确认最终根因，不能下结论说数据库算力就是瓶颈。

**更严谨的回答**

> 当前可以确认的是批处理总耗时超出业务目标，且累计数据库访问时间占比较高。下一步需要继续拆解累计耗时最高的 SQL、执行次数、返回行数和总扫描量，并结合执行计划、连接等待、锁等待、CPU 和 IO 指标判断根因。现阶段不能只凭 JDBC 方法耗时直接认定数据库算力不足。
