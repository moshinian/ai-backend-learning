**评价**

- 评分：7/10
- 回答正确的部分：职责边界基本清晰，符合当前项目事实。
- 主要问题：需要主动解释为什么 Java 不直接做全部能力，以及跨语言调用带来的代价。

**更好的回答**

### 3.3 知识库构建链路

**面试官关注点**

> 用户上传文档后，系统怎么处理？Embedding 和数据库写入分别在哪里做？

**我的现场回答**

> 上传后 Java 做初处理和任务管理，文档解析后切块，调用 Python 进行 Embedding，再由 Java 写入 PostgreSQL 和 pgvector，同时存储章节 ID、chunk ID 等元数据。

**评价**

- 评分：7/10
- 回答正确的部分：链路完整，数据归属基本清楚。
- 主要问题：可以更明确任务状态：PENDING、PROCESSING、DONE、FAILED、RETRY_WAIT，以及哪些失败可重试。

**更好的回答**

> 文件上传后先在 Java 创建索引任务和文档记录，状态为待处理。后台任务解析文档、按策略切 Chunk，并调用 Python 生成 Embedding。Java 拿到向量后，把 chunk 文本、向量、文档 ID、章节 ID、来源位置、模型版本和切分策略一起写入 PostgreSQL / pgvector。解析失败这类数据问题通常不可重试；模型调用超时、服务短暂不可用这类基础设施问题可以退避重试。

### 3.4 RAG 问答链路与证据链

**面试官关注点**

> 问答时怎么召回？证据链如何展示？元数据是否参与模型决策？

**我的现场回答**

> 使用语义匹配和精准词检索做权重排序，召回后结合上下文 Prompt 交给大模型生成答案。向量化时同步存储章节 ID 等元数据，主要用于前端证据链展示，不直接参与大模型决策。

**评价**

- 评分：7/10
- 回答正确的部分：能区分检索、生成和证据链展示；知道 chunk / 章节元数据用于溯源。
- 主要问题：需要稳定区分全文检索、向量检索、Fusion、Rerank；不要把所有排序都叫 Rerank。

**更好的回答**

> 在线问答时先做两路召回：pgvector 负责语义相似，全文检索负责关键词和专有名词。两路结果做分数归一化或 RRF 融合，再选 TopK chunk 组装 Prompt。每个 chunk 入库时绑定文档 ID、章节 ID、位置和来源 URL，所以前端可以展示引用和展开上下文。元数据主要用于过滤、溯源和展示；是否进入 Prompt 要看它是否有助于回答，不能把无关元数据全部塞给模型。

### 3.5 Chunk 切分策略

**面试官关注点**

> 固定窗口之外还有哪些切分方式？业务场景下怎么切？

**我的现场回答**

> 可以用固定窗口加重叠，边界尽量靠近标点断句。小文本可按自然段切分；接口文档等业务结构化内容，应把单个接口、参数、返回值等完整业务单元放在同一个 chunk，并写入语义标签。关联片段召回时优先拼成证据链。

**评价**

- 评分：7.5/10
- 回答正确的部分：能从固定窗口扩展到自然段和业务结构切分。
- 主要问题：对语义切分概念不熟，只能用 Embedding 距离去类比，缺少清晰定义。

**更好的回答**

> 切分策略可以分三类。第一类是通用窗口切分，用最大 token 长度和 overlap 保证上下文不断裂。第二类是结构化切分，比如接口文档按接口、参数、返回值和错误码组织，规则文档按条款、条件和例外组织。第三类是语义切分，用句子或段落的语义相似度变化判断主题边界。实际项目里我会先用结构和窗口兜底，再用离线问答集验证 Recall@K 和引用准确性。

### 3.6 异步任务、超时和重试

**面试官关注点**

> 大模型调用卡住怎么办？文档索引失败如何恢复？哪些异常应该重试？

**我的现场回答**

> 问答请求偏同步，但结果会入库，用户断开后可查看最终结果。文档索引是后台异步任务，状态落库，失败后按 10 秒、20 秒等间隔递增重试。文档解析失败属于不可重试，服务器宕机、外部接口临时失败可以靠重试恢复。

**评价**

- 评分：7/10
- 回答正确的部分：知道问答与索引处理模式不同；知道异常要分类。
- 主要问题：一开始对“异步索引失败重试”的具体场景有卡壳，说明简历表达和实现细节没有完全对齐。

**更好的回答**

> 我会把异常分成三类：第一类是业务或数据不可重试异常，例如文件格式不支持、解析结果为空，直接失败并提示；第二类是外部依赖短暂异常，例如模型 API 超时、Python 服务短暂不可用，进入 RETRY_WAIT 并做指数退避；第三类是未知异常，记录上下文后做有限次数兜底重试，超过阈值进入 DEAD 或人工处理。任务状态、重试次数、错误码和最后错误信息都要落库。

### 3.7 百万级文档与检索扩展性

**面试官关注点**

> 如果文档量到百万级，检索延迟上升怎么办？

**我的现场回答**

> 可以先优化索引和业务流程，兜底方案是分库分表。切分维度可以按用户维度、日期分区，检索时通过时间条件缩小范围。

**评价**

- 评分：6.5/10
- 回答正确的部分：能想到先优化索引和过滤，再考虑分库分表。
- 主要问题：分库分表说得过早；RAG 检索扩展应先讲元数据过滤、向量索引参数、冷热分层、分区召回、TopK 合并和离线评测。

**更好的回答**

> 我会先确认瓶颈是在向量检索、全文检索、Rerank、模型生成还是数据库过滤。数据层先用租户、知识库、权限、生效时间等元数据过滤，避免在全量库上做 TopK；pgvector 侧优化 HNSW / IVFFlat 索引和参数；业务上做冷热分区、归档和按知识库分区召回。只有单库容量、写入维护或隔离要求明显超出时，再考虑按租户或知识库维度拆分，并在应用层做多分区召回和结果合并。

**我的现场回答**

> 询问对方 Agent 是否涵盖工具、上下文及人工确认链路，并确认后端负责业务、Python 负责 Agent 构建。

### 4.2 Redis 常见数据结构

**面试官原问题**

> Redis 底层数据结构 / 常用数据结构有哪些？结构体怎么存？

**我的现场回答**

> 现场回答偏向布尔、整数等底层类型；结构体采用序列化后存入 key-value 的形式。

**评价**

- 评分：3/10
- 回答正确的部分：知道可以把对象序列化成 String value。
- 主要问题：没有先回答 Redis 对外数据类型：String、Hash、List、Set、ZSet、Stream、Bitmap、HyperLogLog、Geo；Hash 与序列化 String 的使用边界不清。

**更好的回答**

> Redis 常用数据类型包括 String、Hash、List、Set、ZSet 和 Stream。对象可以序列化成 String，也可以按字段存成 Hash。String 适合整体读写和缓存完整对象，实现简单；Hash 适合按字段读写、局部更新和减少反序列化成本，但字段结构变化和大对象管理要注意。分布式锁通常用 String，因为只需要一个 key 对应唯一持锁 value 和 TTL。

**需要补充学习**

1. Redis 数据类型和典型工程场景。
2. Hash 与 String 序列化对象的边界。
3. Redis 底层编码可以后置，先把对外数据类型讲清楚。

### 4.3 Redis 分布式锁与锁粒度

**面试官关注点**

> Redis 锁如何避免误删？锁过期怎么办？任务执行时间超过锁超时怎么办？锁应该覆盖整个任务还是只覆盖临界区？

**我的现场回答**

> 用任务唯一 ID 做 key，用机器唯一键或业务键加雪花 ID 做 value，加超时时间防死锁；finally 中删除锁前校验唯一值，防止误删其他服务器的锁。任务状态置为处理中后，后续任务检索不到该数据，避免重复处理。面试官建议锁只保护“查询并变更待执行任务”的临界操作，长耗时业务从锁里剥离。

**评价**

- 评分：6.5/10
- 回答正确的部分：知道 `unique_value` 防误删、TTL 防死锁、状态机可降低重复处理风险。
- 主要问题：没有一开始主动区分两类锁：业务任务抢占锁通常只保护临界区；框架级任务锁如果目标是全周期互斥，就必须覆盖整个任务并续期。

**更好的回答**

> 这里要先区分锁的目标。如果只是防止多个 worker 同时抢同一批业务数据，锁应该尽量短，只覆盖“查询候选任务并把状态改成 PROCESSING”的临界区，之后依靠数据库状态、process_token、心跳和幂等推进长任务。如果是框架级锁，例如同一个定时任务同一时间只允许一个实例运行，那锁的语义就是覆盖整个任务周期，必须配合合理 TTL、看门狗续期、任务超时和异常释放。不能把这两类锁混成一个答案。

### 4.4 后端分层、单测和日志

**面试官关注点**

> HTTP 到 DB 的垂直分层是什么？各层职责是什么？单测测哪里？日志如何串联？

**我的现场回答**

> Controller 作为 HTTP 入口，业务层处理逻辑，Server / Service 层处理事务与查询，DAO / Repository 访问数据库，Job 层处理定时和异步任务。单测主要测 Service 层，通过 Mock 外部依赖。日志记录 API 集成、MQ 交互、请求和响应体，并通过公共方法封装。

**评价**

- 评分：6/10
- 回答正确的部分：基础分层和 Service 层单测策略能讲出来。
- 主要问题：日志链路没有稳定讲出 TraceId / requestId 如何跨 HTTP、MQ、外部 API 和 Job 串起来；Server / Service 表达也需要统一术语。

**更好的回答**

> 分层上 Controller 负责协议入口、参数校验和响应转换；Service 负责业务编排、事务边界和状态流转；Repository / DAO 负责数据访问；Client 负责外部系统调用；Job / Consumer 负责异步入口。单测重点放在 Service 的业务分支、状态流转和异常处理，外部依赖用 Mock。日志上每次请求生成 traceId，调用外部 API、发送 MQ、消费 MQ 和执行 Job 时都把 traceId、bizId、messageId、requestId 透传到结构化日志里，这样才能串起跨服务链路。

6. 工作节奏：对方描述为 965，使用 GLM 等性价比较高的模型。

## 6. 跨题错误模式

1. **事实边界要更主动**：个人 RAG 项目不能讲成生产项目，但也不能只说“没上线”。要讲清“真实痛点、已实现能力、未验证边界、下一步指标”。
2. **Redis 基础必须补**：常见数据类型、Hash 与 String、锁数据结构、使用场景需要马上补齐。
3. **锁粒度先问目标**：业务抢占锁、框架级任务锁、数据库状态机解决的问题不同，不能混答。
4. **Agent 不能只讲组件**：要讲任务状态、决策、工具执行、观察、再决策和人工确认。
5. **日志链路不能只说打日志**：需要能讲 traceId、bizId、messageId、外部 requestId、结构化日志和告警。
6. **简历表达要和实现细节对齐**：写了“异步索引失败重试”，就必须能立刻说出具体状态、失败分类和重试策略。
