## 问题 3：你怎么理解 Harness / Skill / MCP / API

问题：

> 你怎么理解 Harness、Skill、MCP 和 API 的关系？

现场回答或目标回答：

```text
我的理解是，Harness 是承载 AI 应用运行和工具调用的工程层，负责把模型、上下文、工具、执行状态、安全策略和观测能力组织起来。

API 是外部系统最基础的能力暴露方式，例如 ERP 查询订单、创建单据、审批流查询。Skill 更像面向 Agent 或业务场景封装过的一组能力，它不只是一个接口，还包括参数 schema、说明、权限、错误处理和业务语义。MCP 则提供一种更标准化的协议，让模型或 Agent 能用统一方式发现和调用工具、资源和上下文。

工程上我会把它们理解成三层：底层是 ERP 或第三方系统 API；中间层做 Adapter，处理鉴权、字段映射、幂等、超时、审计；上层再暴露成 Skill 或 MCP Tool 给 Harness / Agent 调用。
```

评价：

- 回答要体现“标准化接入”，不是只背 MCP 定义。
- 重点是 API -> Adapter -> Skill / MCP Tool -> Harness / Agent。

## 问题 4：ERP 工具接入怎么设计

问题：

> 如果让你把一个 ERP 能力接入 Agent 工具体系，你怎么设计？

现场回答或目标回答：

```text
我会先把 ERP 能力分级。查询类工具风险较低，例如查询客户、订单、库存、审批状态；写操作风险更高，例如创建单据、修改价格、发起审批、删除数据。

工程链路上，第一步定义工具 schema，包括工具名、用途、参数、返回值、错误码和权限要求。第二步做 Adapter 层，对接 ERP API，处理鉴权、租户、字段映射、超时、重试和错误转换。第三步接入 Harness，让 Agent 能基于工具说明生成调用参数。第四步做治理，包括限流、审计、traceId、调用日志、指标监控和灰度发布。

对于高风险写操作，我不会让模型直接执行。模型只能生成建议或待确认 action，后端落库为 WAITING_CONFIRMATION，用户确认后再由受控节点校验权限、状态和幂等键，然后执行真实 ERP API。
```

评价：

- 这是本岗位核心问题之一。
- 必须说清“查询工具”和“写操作工具”的风险差异。

改进版：

- 可以主动补一句：ERP 对接最难的不是调通接口，而是字段语义、权限、租户隔离、幂等和异常补偿。

## 问题 5：Agent 工具调用如何保证安全

问题：

> Agent 调用外部系统工具，怎么保证安全和可靠？

现场回答或目标回答：

```text
我会把安全和可靠拆成几层。

第一是工具注册治理。每个工具必须有清晰的 schema、权限范围、风险级别、超时、限流、审计字段和负责人。

第二是调用前校验。Harness 或后端要校验用户身份、租户、角色、工具权限和参数合法性，不能只依赖 Prompt 约束模型。

第三是执行控制。查询类工具可以直接执行，但写操作要走审批或 human-in-the-loop；执行时带 requestId、traceId 和 idempotencyKey，避免重复调用。

第四是失败处理。外部系统超时要区分未知结果和确定失败，不能简单重试导致重复写；需要状态机、补偿任务、死信或人工处理入口。

第五是审计和回放。保存用户问题、模型决策、工具参数、调用结果、耗时、错误码和最终输出，方便排查和合规。
```

评价：

- 关键句：Prompt 不是安全边界，后端权限和状态才是安全边界。
- 这个回答能连接 LangGraph interrupt 和 Java DB 业务事实边界。

改进版：

- 如果被问 LangGraph，回答 checkpoint 负责运行现场，数据库负责业务事实和审批结果。

后续任务：

- 关联 `interview/ai-application-questions.md` 的高风险工具执行边界。

## 问题 6：服务治理和高可用怎么做

问题：

> 你怎么设计统一服务治理体系，保障外部生态接入的高可用和可靠性？

现场回答或目标回答：

```text
我会按接入前、运行中、故障后三个阶段设计。

接入前要做标准化：统一 API / Skill 规范、鉴权方式、错误码、超时、重试策略、版本号、灰度规则和文档模板。

运行中要做治理：服务注册发现、限流、熔断、隔离、超时控制、重试退避、调用链追踪、日志、指标和告警。对下游 ERP 或第三方 API，调用方线程池和并发数要受下游容量约束，不能无限加线程。

故障后要能恢复：可重试失败进入延迟重试或 MQ，未知结果进入对账或人工处理，写操作用幂等键防止重复，重要调用保留审计和回放数据。

如果是 AI 工具生态，还要额外关注工具版本、Prompt / Tool schema 版本、模型版本和灰度回滚，避免 Agent 调用到不兼容工具。
```

## 问题 12：工具调用失败、超时或重复执行

问题：

> Agent 调用工具失败、超时或者重复执行怎么办？

现场回答或目标回答：

```text
工具调用失败要先分类，不能统一重试。

查询类工具如果遇到网络抖动或短暂超时，可以做有限次数重试和退避。写操作要谨慎，比如创建单据、修改状态、发起审批，超时后可能是外部系统已经成功但没返回，所以不能盲目重试，要通过业务单号或 idempotencyKey 查询外部状态，再决定补偿、重试或人工处理。

重复执行要在设计阶段控制。写操作必须带业务唯一键或 idempotencyKey，并结合后端状态机保证同一个 action 只能执行一次。

对于参数错误、权限不足、业务状态冲突这类不可恢复错误，就不重试，直接记录 traceId、错误码和上下文，返回明确原因或转人工。
```

评价：

- 核心是“写操作超时是未知结果”，不能简单当失败。

改进版：

- 结合 ERP 入账场景时，强调查状态、对账和补偿。

后续任务：

```text
我们主要靠三层防重复入账：业务唯一键、本端状态机、下游幂等。

本端会根据结算单号、账期、业务类型等字段生成唯一业务键，并在数据库做唯一约束。状态机上区分推送状态和入账状态，推送状态包括未推送、推送中、推送成功、推送失败。后台任务只处理未推送或可重试失败的数据，并通过条件更新把状态抢占成推送中，只有抢占成功的实例才能调用 ERP。

调用 ERP 时也会带外部请求号或业务唯一键，让下游做幂等。同步 API 成功只代表推送受理，不一定代表最终入账成功；最终结果要看 MQ 回写、查询或对账。若超时没有结果，不能盲目重推，要先查 ERP 状态，确认后再补偿或重试。
```

评价：

- 关键词：唯一业务键、条件更新抢占、推送状态、入账状态、下游幂等、超时查状态 / 对账。

改进版：

- 不要把同步推送成功说成最终入账成功。

后续任务：

- 面试后记录是否被问到 ERP 入账细节。

## 问题 14：Agent 诊断订单未入账

问题：

> 如果用户问 Agent：“帮我查一下这笔订单为什么没有入账，并给出修复建议”，你会怎么设计执行流程？

现场回答或目标回答：

```text
我会把它设计成受控 Agent 诊断流程，而不是让模型直接改业务数据。

前端请求到后端后，后端补齐用户、租户、权限、订单号、traceId，再传给 AI Gateway。AI Gateway 根据权限加载可用只读工具，比如查本地订单、结算单、推送状态、ERP 入账状态、MQ 回写记录和异常任务记录，再交给 Agent Runtime。

Agent 根据问题选择工具并生成参数，但执行前要做 Schema、权限和风险校验。诊断时按链路逐层排查：订单是否存在，是否生成结算单，是否推送 ERP，ERP 是否受理，MQ 是否回写，本端和 ERP 状态是否一致。

最后 Agent 输出诊断结论和修复建议。如果涉及重推、补偿、修改状态这类高风险动作，只生成待确认 action，人工确认后由后端受控节点带 idempotencyKey 执行，并记录 traceId 做审计和回放。
```

评价：

- 业务链路必须落到订单、结算单、推送状态、ERP 状态、MQ 回写，不能只泛讲 Agent loop。

改进版：

- 把“LLM 决策”改成“模型选择工具、生成参数和规划下一步；工具执行和业务状态变更由后端受控执行”。

## 问题 15：RAG 评测与 Hybrid / Rerank

问题：

> 你怎么评估 RAG 效果？Hybrid Retrieval 和 Rerank 有什么区别？

现场回答或目标回答：

```text
我会把 RAG 评测拆成四层。第一是检索召回，看 Recall@K，判断正确证据有没有进入 TopK。第二是排序质量，看 MRR、NDCG，如果正确 Chunk 召回了但排得靠后，就说明 Fusion 或 Rerank 有问题。第三是生成质量，看答案正确性、faithfulness、引用准确性和拒答能力。第四是工程指标，比如延迟、错误率、成本和索引任务成功率。

Hybrid Retrieval 作用在召回层，解决的是“能不能把相关证据召回来”。向量检索负责语义相似，关键词或 BM25 负责订单号、错误码、专有名词这类精确匹配。

多路召回后需要 Fusion，比如 RRF 或归一化加权，把不同路径的候选合并。Rerank 是召回后的精排，它不从全库找新文档，而是对已召回候选重新打分。常见 cross-encoder 会把 query 和 chunk 作为一对输入模型，输出相关性分数。
```

评价：

- 记住：Hybrid 解决召回覆盖，Fusion 解决多路合并，Rerank 解决候选精排。

改进版：

- 如果正确文档被召回但排最后，先回放定位 Fusion / Rerank，再短期规则特征、中期评测集、长期微调或换模型。

后续任务：

- 继续回到 `BL-013` 深挖私有语料排序失效。

## 问题 16：语义切分怎么做

问题：

> 除了段落和窗口，有没有办法做语义切分？

现场回答或目标回答：

```text
我当前项目实际落地的是固定窗口、overlap 和自然边界兜底。语义切分我理解为后续优化方向。

工程实现上，可以先把文档拆成句子或自然段，再对每个单元或相邻句子组生成 embedding，计算相邻单元之间的 cosine similarity。如果相邻段落的语义距离明显变大，就说明主题可能发生切换，可以作为候选切分点。

但它不能只靠 embedding 距离。还要加 min_tokens、max_tokens、overlap 约束，避免切得太碎或太长；同时要保护标题、表格、接口说明这类结构化块。最终用 Recall@K、MRR、引用准确性和回答忠实度来验证切分效果。
```

评价：

- 不能包装成已完整落地。
- 关键词：基础单元、相邻单元组、embedding 相似度、候选切分点、token 约束、结构保护。

改进版：

- 不要说“直接让大模型切”；大模型辅助切分只能作为离线增强方案。

后续任务：

- 后续将语义切分沉淀到正式 RAG 项目笔记。

## 问题 17：表格怎么 Chunk

问题：

> PDF、Word、CSV 中的表格都不是固定结构，表格怎么 chunk？

现场回答或目标回答：

```text
表格不能按普通文本直接固定窗口切，因为表格语义来自行、列、表头、合并单元格、单位和脚注。成熟做法是先做表格结构抽取，再按结构 chunk。

如果是 CSV / Excel 这类规则表，我会把每一行转成 key-value block，比如“订单号: A001，状态: 失败，原因: ERP 超时”，再按 token 预算合并多行，同时每个 chunk 都带上表名、表头、单位和来源。

如果是 Word / PDF 表格，先用解析工具拿到行列、表头、rowspan / colspan、页码和坐标。小表可以整体作为一个 chunk；大表按行组 chunk；宽表按列族拆；交叉表要把行头、列头和值转成事实句。

另外会保留多种表示：原始 HTML / Markdown 用于展示，结构化 JSON 用于精确字段，row-level KV 用于检索，table summary 用于表级召回。最后用表格问答评测集看 row recall、cell accuracy、引用准确性和答案正确性。
```

评价：

- 不能说“把表格转 Markdown 后固定窗口切”就结束。
- 核心是先抽结构，再按表格类型处理。

改进版：

- 当前项目没有完整表格解析生产化落地，表达为成熟优化方向。

后续任务：

- 面试后若继续暴露表格解析短板，优先把它拆成 `BL-013` 下的独立子任务。
