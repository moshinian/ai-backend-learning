## 问题 2：RAG 系统怎么做

问题：

> 你介绍一下企业知识库 RAG 系统是怎么做的？

现场回答或目标回答：

```text
RAG 系统主要解决的是企业内部知识检索和问答的问题。

整体上我会把它分成两条链路：离线索引链路和在线问答链路。

离线索引链路包括文档上传、文档解析、Chunk 切分、Embedding 向量化、向量入库，同时保存文档、分块、索引任务这些元数据。

在线问答链路是用户提出问题后，先对问题做向量化，再到向量库里做相似度匹配，召回相关 Chunk，然后把用户问题和召回内容组装成 Prompt，调用大模型生成回答，最后保存会话和引用来源。

工程上我重点关注几个问题：索引任务可能失败，所以需要任务状态、失败重试和异常修复；回答要能够溯源，知道依据来自哪些文档片段；金融财经类场景要考虑权限隔离和审计；如果线上效果不好，要能定位问题到底出在文档解析、Chunk、召回、重排、Prompt，还是模型生成。
```

评价：

- 主线清晰，能体现不是只调 API。
- 排查链路顺序要稳定：解析 / Chunk -> 召回 -> 重排 -> Prompt -> 模型生成。

## 问题 3：Chunk 怎么切

问题：

> Chunk 为什么要切？怎么切？切大切小有什么影响？

现场回答或目标回答：

```text
Chunk 主要解决两个问题：一是大模型上下文窗口有限，不能把整篇文档都塞进去；二是检索需要合适粒度，如果直接按整篇文档检索，召回内容太粗。

我的做法是按固定窗口切分，同时在相邻 Chunk 之间保留一定 overlap，避免答案刚好跨越边界时信息断裂。切分末端会留一定调整空间，让切分点尽量落在句号、换行、标题这类自然边界上。

Chunk 太大，召回内容噪声多，会污染 Prompt；Chunk 太小，语义不完整，容易丢上下文。所以 Chunk 大小本质上是在召回精度和上下文完整性之间做平衡。
```

评价：

- 不要只说固定窗口，要先说明为什么切。

改进版：

- 具体大小不要报死数，按文档类型、模型上下文和评估结果调整。

后续任务：

- 后续可补充 parent-child retrieval、按标题层级切分等优化方向，但今晚不要包装成已实现。

## 问题 4：Embedding 和向量检索

问题：

> Embedding 和向量检索解决什么问题？为什么不用关键词搜索？

现场回答或目标回答：

```text
Embedding 的作用是把用户问题和文档 Chunk 转成向量表示，让它们可以在同一个语义空间里比较相似度。

向量检索的作用是根据这个相似度，从知识库里召回和用户问题语义相关的 Chunk。用户问法和文档原文用词不完全一致时，只要语义接近，也有机会召回。

关键词搜索主要依赖词面匹配，优点是精确、可解释，适合专有名词、编号、证券代码、产品名称这类场景。问题是问法和文档词不一致时可能召不回来。

所以生产上更稳的是混合检索：关键词检索保证精确命中，向量检索补语义召回，后面再通过重排或融合策略选出更相关的片段。
```

评价：

- 不要贬低关键词搜索，金融证券场景里实体、编号、代码非常依赖精确匹配。

改进版：

- 如果面试官问 hybrid search，重点讲 BM25 / 关键词 + 向量召回 + 融合 / rerank。

后续任务：

- 继续补 hybrid search 的工程实现边界。

## 问题 5：RAG 效果不好怎么排查

问题：

> 召回结果不准，或者 RAG 回答效果不好，你怎么排查？

现场回答或目标回答：

```text
我会先把 RAG 效果问题拆成两类：是检索没拿到正确证据，还是拿到了证据但模型没有正确使用。

第一步做链路回放，记录 requestId 下的用户原问题、改写后的问题、召回 Chunk、重排结果、最终 Prompt 和模型回答。

如果怀疑召回不准，就看 Recall@K：正确答案所在的 Chunk 有没有进入 TopK。如果没有进入，排查 Chunk 切分、Embedding 模型、向量库参数、相似度阈值，必要时引入 hybrid search。

如果正确 Chunk 已进入 TopK，但排序靠后，重点看重排或融合策略。

如果正确证据已经进入最终 Prompt，但模型回答仍然不对，问题更可能在 Prompt 结构、上下文噪声、模型参数或模型能力本身。
```

评价：

- 重点是“正确证据丢在哪一层”，不是先猜换模型或换 embedding。

改进版：

- 牢记：召回前、召回后、重排后、Prompt 中、模型生成阶段。

## 问题 7：权限隔离

问题：

> 不同用户权限不同，RAG 检索怎么做权限隔离？

现场回答或目标回答：

```text
权限隔离要尽量放在检索阶段，而不是等大模型生成完再过滤。

可以从租户、知识库、文档，甚至 Chunk 维度保存权限信息，比如 tenant_id、knowledge_base_id、department_id、role、permission_code、owner、document_status 等字段。

用户发起 RAG 查询时，后端先根据身份解析可访问范围，然后在向量检索时带上过滤条件，限制候选 Chunk。这样无权限内容不会进入召回结果，也不会进入 Prompt。

最终回答保存引用来源时，也要保存用户、问题、命中的文档 / Chunk、权限范围和时间，方便后续审计和问题追踪。
```

评价：

- 关键句：权限不能只靠 Prompt，必须在检索候选集阶段控制。

改进版：

- 如果追问性能，回答按租户 / 知识库缩小候选集、分区、分表、分索引。

## 问题 8：降低幻觉和可信回答

问题：

> RAG 怎么降低幻觉？怎么保证回答可信？

现场回答或目标回答：

```text
RAG 降低幻觉不能只靠一句 Prompt 约束，我会从三层做。

第一层是证据输入。召回内容要准确，Chunk 数量和长度要合适，不能把大量噪声塞进上下文；如果多个版本冲突，要优先使用有效版本，或者提示存在版本冲突。

第二层是 Prompt 约束。要求模型基于给定资料回答，不要编造；如果资料里没有依据，就说明无法从当前资料确认。

第三层是输出可追溯。回答里保留引用来源，能追到具体文档和片段。这样用户可以回看证据，系统也能定位问题是召回错了、证据冲突，还是模型没有按证据回答。
```

评价：

- 金融场景下可信度来自证据链、权限、版本、审计，不来自模型自信。

改进版：

- 如果模型仍胡说，可补 faithfulness 检查或后处理校验，但不要把它当根本方案。

后续任务：

- 面试后根据真实追问决定是否补 `mistakes/`。

## 问题 9：索引任务失败

问题：

> 文档索引任务失败了，怎么设计状态、重试和异常修复？

现场回答或目标回答：

```text
我会把文档索引设计成明确的任务状态机，而不是只靠一个失败标记。

任务状态可以包括 PENDING、PROCESSING、SUCCESS、RETRYABLE_FAILED、DEAD。用户上传文档后先创建索引任务。后台 worker 获取任务时，把状态改成 PROCESSING，同时记录 process_token、开始时间、重试次数和当前处理节点。

失败后先区分类型。模型服务超时、向量库短暂不可用可以重试；文件格式不支持、文档损坏、解析为空不应该一直重试。

可重试失败由定时任务或 MQ 延迟重试，超过阈值后进入 DEAD 或人工处理状态。

多实例部署时，获取任务要结合数据库状态更新、process_token，必要时配合 Redis 锁。写 Chunk 和向量也要幂等，避免半成功导致新旧数据混在一起。
```

评价：

- 要突出状态机、重试分类、并发控制、幂等。

改进版：

- 不要只说“失败后定时任务重试”，否则会被追问重复处理和半成功。

后续任务：

- 与 `BL-002` Redis 锁粒度继续关联复盘。

## 问题 10：金融证券场景风险

问题：

> 如果 RAG 用于证券投研或客户问答，最大的风险是什么？怎么控制？

现场回答或目标回答：

```text
风险主要有几类。

第一是权限泄露。不同用户、部门、客户能访问的资料不同，必须在租户、知识库、文档或 Chunk 维度做权限过滤，且过滤发生在召回阶段。

第二是错误或误导性回答。金融场景里模型不能随便给确定性投资建议，Prompt 要约束回答边界，资料不足时拒答或提示无法确认。

第三是依据不可追溯。回答必须带引用来源，能追到具体文档和片段。

第四是合规和审计。要记录用户、时间、问题、召回片段、引用文档、模型回答、模型版本等信息，方便事后审计和问题回放。

金融场景下，RAG 系统宁可保守拒答，也不能无依据地编造一个看起来专业的答案。
```

评价：

- 不要只讲权限，要覆盖误导性回答、依据追溯、合规审计。

改进版：

- 如果追问客户问答，要强调非投资建议、合规话术、人工兜底。

## 问题 12：Redis / MQ / Spring / FastAPI

问题：

> 大模型应用里为什么还需要 Redis / MQ？Spring 和 FastAPI 怎么分工？

现场回答或目标回答：

```text
Redis 可以用于热点配置或元数据缓存、限流、临时状态，也可以在索引任务中辅助多实例并发控制。但我不会只依赖 Redis 锁保证任务只处理一次，还会结合数据库任务状态、process_token、重试次数和幂等设计。

MQ 主要解决异步解耦和削峰。比如用户上传文档后，不同步完成解析、Chunk、Embedding 和入库，而是创建索引任务，由后台 worker 或 MQ 消费。MQ 也能用于失败重试、死信队列、外部系统通知和补偿。

Spring / Java 负责业务接口、权限隔离、租户、文档和 Chunk 元数据、索引任务状态、日志审计、会话记录、异常修复和任务管理。

FastAPI / Python 更适合封装 AI 能力，比如 LLM 调用、Embedding 模型调用、Rerank、Prompt 处理，或接 LangChain / LangGraph 生态。

总结就是：Java 是业务和状态权威，Python 是 AI 能力适配层。
```

评价：

- 要把 Redis / MQ 讲到大模型应用落地，不要泛泛背中间件。

改进版：

- 线程池、下游容量、重试退避和异步任务可以作为追问扩展。

## 问题 14：项目未上线和能力证明

问题：

> 你的 RAG 项目有没有真实生产上线？效果怎么评估？

现场回答或目标回答：

```text
这个 RAG 项目没有在真实生产线上线，我不会把它包装成生产经验。
```
