### 问题 4：Nebula 和 MySQL 的区别

问题：

> Nebula 和 MySQL 有什么区别？

现场回答或目标回答：

- MySQL 是关系型数据库，适合结构化表、事务、约束、聚合和常规业务 CRUD。
- NebulaGraph 是图数据库，核心数据模型是点、边和属性，适合多跳关系查询、路径发现和关系网络。
- 选择依据不是谁性能更高，而是访问模式：按主键、条件和关联表查询优先考虑关系模型；大量多跳关系遍历更适合图模型。
- 事务、约束、查询语言、数据建模和运维生态不同，不能用图数据库替代所有业务主库。

评价：

- 如果没有真实使用 Nebula，必须明确“了解原理或做过调研”，不能说成项目落地。
- 面试官可能继续追问图数据库在知识图谱或 RAG 中的应用。

改进版：

- 补一张“数据模型、查询方式、事务能力、典型场景、边界”的对比表。

后续任务：

- 确认简历或岗位描述为什么出现 Nebula，再决定准备深度。

### 问题 5：Spring 依赖注入有什么意义

问题：

> Spring 依赖注入有什么意义？

现场回答或目标回答：

- 核心不是“少写 `new`”，而是把对象创建、依赖装配和生命周期交给容器管理。
- 业务类依赖接口而不是具体实现，可以降低耦合，方便替换实现、测试和统一代理增强。
- 依赖注入本身不自动保证设计合理；依赖过多仍说明类职责可能过重。

评价：

- 需要从对象协作和可替换性解释，而不是只背 IOC 定义。

改进版：

- 准备构造器注入为什么优于字段注入的追问。

后续任务：

### 问题 6：Bean 创建时机、单例和原型

问题：

> Spring Bean 什么时候创建？单例模式和原型模式有什么区别？

现场回答或目标回答：

- 默认非懒加载单例 Bean 通常在容器启动、创建 `ApplicationContext` 的过程中实例化。
- `@Lazy` 单例在首次获取或首次依赖解析时创建；prototype 通常在每次向容器请求时创建。
- singleton 表示同一个容器中同一 Bean 定义通常只有一个实例，不等同于 GoF 单例模式的所有实现细节。
- Spring 只完整管理 prototype Bean 的创建和初始化，销毁阶段通常需要使用方自行处理。

评价：

- “Bean 都在启动时创建”过于绝对。
- 需要区分作用域、懒加载和容器类型。

改进版：

- 已补充 Bean 生命周期主线：实例化、属性填充、Aware、前置处理、初始化、后置处理、必要时生成代理、销毁。
- 已确认 prototype 通常只由 Spring 完成创建和初始化，销毁阶段及外部资源释放由使用方负责。

后续任务：

- 进入 `BL-005` 的第一组口述验证。

### 问题 7：Java 集合区别

问题：

> HashMap、ConcurrentHashMap、Hashtable 有什么区别？

现场回答或目标回答：

- HashMap 非线程安全，允许一个 null key 和多个 null value。
- Hashtable 是早期线程安全实现，主要方法使用同步控制，不允许 null key/value，接口和并发设计较旧。
- ConcurrentHashMap 面向并发访问，不允许 null key/value；现代实现通过 CAS、桶级同步等方式降低竞争，读操作通常具有更好的并发性。
- 选择时先判断是否存在共享并发写，而不是看到多线程就机械使用 ConcurrentHashMap。

评价：

- 不能只回答“一个线程安全、一个不安全”。
- 可能继续追问 JDK 7 与 JDK 8 的结构差异、扩容和 `size()`。

改进版：

- 已补 ConcurrentHashMap 的 Java 8 以后主线：读取通常不加互斥锁、空桶 CAS、冲突桶局部 `synchronized`、协作扩容和分散计数。
- 已确认线程安全容器不等于多步逻辑原子化，`containsKey` 再 `put` 应根据语义改用 `putIfAbsent` 等原子方法。

后续任务：

- 完成一次结构图口述。

### 问题 8：`@Transactional` 事务失效

问题：

> `@Transactional` 有哪些常见失效场景？

现场回答或目标回答：

- 核心判断是调用是否经过 Spring 事务代理，以及异常是否触发回滚。
- 常见场景包括同类内部自调用绕过代理、方法不可被正确代理、对象不是 Spring Bean、异常被捕获未继续抛出、默认不回滚 checked exception、传播行为或数据源配置不符合预期、异步或新线程越过事务上下文。
- 数据库或存储引擎本身不支持事务，以及最终没有实际事务边界，也会导致预期不成立。

评价：

- 不要把所有问题都笼统说成“注解没生效”。

改进版：

- 用“代理边界、异常边界、线程边界、数据库边界”四类组织答案。
- 真实例子使用对账辅账事故：事务方法同类自调用，状态更新与辅账生成没有可靠落在统一事务边界；拆分独立 Spring Service 后恢复事务一致性。

后续任务：

- 结合一个真实 Service 调用链复述。

### 问题 9：生产环境事故如何排查

问题：

> 线上出现异常、接口变慢或服务不可用，你怎么排查？

- 新建独立 Spring Service，迁移一致性方法并增加事务注解；测试验证异常回滚，修复后未再发生同类问题。

- 已有真实现象、影响范围、证据链、代码根因、修复和结果，是当前最强生产事故案例。
- 不能把 `Could not roll back JDBC transaction` 直接解释为数据库已经部分提交；它只说明 JDBC 回滚动作异常，最终数据状态需要核对。

改进版：

- 口述时按“用户现象 -> 影响范围 -> 日志证据 -> 代码根因 -> 修复验证”推进，先讲事实，再解释 Spring 代理和 JDBC 回滚边界。

### 问题 10：SQL 如何优化

问题：

> 慢 SQL 怎么分析和优化？

现场回答或目标回答：

- 主讲真实 N+1 案例：流水匹配订单每分钟不足 1000 笔，但 1 秒阈值的慢 SQL 日志没有记录。
- 代码排查发现循环内逐笔查询；单条 SQL 不慢，但累计数据库往返严重。
- 将流水按 1000 笔批量查询订单，按支付流水号构建 Map 做内存匹配；单批查询不足 1 秒，整体达到 100 万笔 8 分钟以内的业务基线。
- 支付流水号索引原本已经存在，贡献点是定位访问模式和完成批处理改造，不声称本人新增索引。

评价：

- 该案例可以说明 SQL 优化不只看慢日志和加索引，还要看调用次数、数据库往返和整体访问模式。

改进版：

- 口述时主动说明“没有慢 SQL 不代表数据库访问没有性能问题”，把单条耗时与累计调用次数分开分析。

后续任务：
